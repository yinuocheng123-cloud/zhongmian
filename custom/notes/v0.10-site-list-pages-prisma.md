# 版本记录：v0.10-site-list-pages-prisma

## 1. 本轮目标

本轮目标是把中眠网前台三条主栏目页从 demo 数据切换到真实 Prisma 数据：

- `/knowledge`
- `/terms`
- `/brands`

本轮重点不是继续补页面样式，而是验证前台主干路径是否已经成立：

- 列表页读取真实已发布数据
- 列表页可以进入真实详情页
- 前台不会暴露未发布内容

## 2. 背景与上下文

本轮承接以下记录：

- `v0.9-site-detail-pages-prisma`
- `v0.8-admin-crud-workflow`

承接关系很明确：

- `v0.8` 先打通后台真实录入、保存、切状态、记录过程
- `v0.9` 再打通三类详情页的真实 Prisma 展示
- 本轮继续补齐三类列表页，让“列表页 -> 详情页”整条前台访问主链路成立

本轮开始前的状态是：

- `/knowledge`、`/terms`、`/brands` 仍然以 `demo-data` 为主
- 虽然详情页已经接上真实数据，但栏目页还是 demo 入口
- 因此前台访问路径还没有真正完成闭环

## 3. 问题分析

表面现象是：栏目页已经存在，视觉结构也已经确认。

真正的问题是：

- 栏目页并没有消费后台真实发布的数据
- 用户从栏目页进入站点时，仍然感受到 demo 站点而不是真实内容系统

本轮最核心的矛盾不是“列表页好不好看”，而是：

- 前台主干路径到底是不是基于真实数据运行

需要优先解决的点有四个：

1. 三条栏目页都必须改为读取 Prisma 数据
2. 都必须只展示 `PUBLISHED` 数据
3. 都要能处理空列表、搜索无结果和数据库不可用
4. 三个栏目页仍然要保持不同角色，不能长成同一种列表模板

## 4. 候选方案比较

### 方案 A：直接在三个栏目页里分别写 Prisma 查询

核心思路：

- 在 `/knowledge`、`/terms`、`/brands` 页面内部各自写列表查询

优点：

- 改动路径短
- 实现快

缺点：

- 查询逻辑会再次分散
- 数据库不可用兜底会重复
- 后续接 metadata、sitemap、列表筛选时难以继续复用

结论：

- 不选

### 方案 B：新增前台列表查询服务层

核心思路：

- 新增 `features/site/lists/server.ts`
- 集中封装三类前台公开列表查询
- 统一处理只读 `PUBLISHED`、最小搜索和无数据库兜底

优点：

- 查询逻辑集中
- 页面层更专注于角色表达和渲染
- 可直接为后续 SEO / sitemap / 聚合优化复用

缺点：

- 需要多维护一个服务层文件
- 前期需要把三条栏目页的差异化查询先梳理清楚

结论：

- 本轮采用

## 5. 最终决策

本轮最终采用“前台列表查询服务层 + 三个动态栏目页”的方式。

关键决策如下：

1. 所有栏目页不再使用 `demo-data` 作为主数据源
2. 所有公开查询统一只消费 `workflowStatus = PUBLISHED`
3. 所有栏目页都保留自己的角色：
   - `/knowledge`：问题入口 / 知识入口
   - `/terms`：术语目录 / 概念入口 / 知识节点
   - `/brands`：行业名录 / 信任展示 / 品牌入口
4. 搜索先只做最小可用版本，不提前做复杂筛选
5. 数据库未配置时，不回退 demo 数据，而是明确提示“真实列表未就绪”

## 6. 具体实现

### 6.1 新增前台列表查询层

新增文件：

- `features/site/lists/server.ts`

该文件负责：

- `getPublishedKnowledgeList`
- `getPublishedTermsList`
- `getPublishedBrandsList`

统一能力：

- 数据库不可用兜底
- 最小搜索
- 数量限制
- 只展示已发布数据

### 6.2 `/knowledge` 接入真实 Prisma 数据

修改文件：

- `app/(site)/knowledge/page.tsx`

查询模型：

- `Content`

查询条件：

- `workflowStatus = PUBLISHED`
- `contentType in [ARTICLE, KNOWLEDGE, TOPIC, GUIDE, REPORT, THINK_TANK]`

已接字段：

- `title`
- `slug`
- `summary`
- `contentType`
- `publishedAt`
- `updatedAt`
- `categories`
- `tags`

搜索已接：

- `q`
- 检索字段：`title / slug / summary`

当前限制：

- 固定最多展示 9 条

### 6.3 `/terms` 接入真实 Prisma 数据

修改文件：

- `app/(site)/terms/page.tsx`

- `Term`

- `workflowStatus = PUBLISHED`

- `slug`

搜索已接：

- `q`

## 7. 本轮优点

相比之前，本轮带来的直接收益有：

1. 三条前台栏目页已经不再依赖 demo 主数据
2. “列表页 -> 详情页” 的整条前台主路径已经打通
3. 前台公开访问逻辑与详情页保持一致，统一只消费已发布数据
4. 查询层已经开始形成前台可复用的数据访问边界
5. 三个栏目页依然保留各自角色，没有被统一成普通内容列表模板

## 8. 本轮缺点与代价

这轮也保留了几个明确妥协：

1. 仍未做复杂筛选，只做了最小搜索
2. 仍未做分页系统，只先用固定数量限制
3. `/knowledge` 目前按内容类型过滤“知识类内容”，还没有进一步按知识频道配置细分
4. 当前本地没有 `DATABASE_URL`，因此运行时只能看到“真实列表未就绪”提示，不能直接做本地真实联调
5. 词条页和品牌页的分类统计当前只统计公开数据，不涉及更复杂的聚合维度

这些代价是当前阶段可接受的，因为本轮目标是先打通前台主路径，而不是一次做完整搜索与聚合系统。

## 9. 验证与结果

本轮实际执行了以下验证：

1. `npm run prisma:generate`
2. `npm run typecheck`
3. `npm run build`
4. `npm run lint`

验证结果：

- 三个前台栏目页均已通过类型检查和构建
- 构建输出中可见：
  - `/knowledge`
  - `/terms`
  - `/brands`
- 三个栏目页均已变为动态服务端渲染路由

额外观察：

- 当前环境中 `DATABASE_URL` 为空
- 因此页面运行时会进入“真实列表未就绪”兜底，而不是展示 demo 数据

## 10. 本轮结论

本轮最重要的结论是：

中眠网前台主干访问路径已经从 demo 结构推进到了真实数据结构。

当前已经成立的链路是：

- 后台录入 `Content / Term / Brand`
- 后台通过发布流转切到 `PUBLISHED`
- 前台列表页读取真实公开数据
- 前台详情页读取真实公开详情

应默认继承的经验有：

- 前台栏目页和详情页必须共用同一套“只公开已发布内容”的逻辑
- 没有数据库时宁可明确提示未就绪，也不要退回 demo 数据制造假象
- 栏目页的数据真实化，不等于放弃栏目角色表达

## 11. 下一轮建议

下一步最值得进入的方向有三个：

1. SEO / metadata / sitemap
   - 收益高，风险低，当前查询层已经为这件事做好基础
2. 前台列表页的进一步筛选与聚合优化
   - 可以继续增强知识入口、词条目录和品牌名录的结构感
3. AI 编辑部基础工作流
   - 现在前后台主路径已经基本成立，后续可以开始把编辑工作流接入真实内容生产

如果只选一个，我会优先推荐：

- `SEO / metadata / sitemap`

因为现在前台主路径已经打通，最顺的下一步就是把这套真实公开内容结构继续变成搜索引擎友好的站点能力。
