# 版本记录：v0.22-knowledge-dev-timeout

## 1. 本轮目标

本轮目标是定位并修复 `/knowledge` 页面在本地开发态长时间无响应的问题。

这一轮属于：
- Bug 修复
- 本地运行稳定性排查

优先级高的原因是：
- 首页与后台登录页已经可访问
- 当前唯一阻塞点是前台知识列表页
- 如果继续误判为 Prisma 慢查询，会把时间浪费在错误方向

## 2. 背景与上下文

本轮承接以下记录：
- `custom/notes/v0.10-site-list-pages-prisma.md`
- `custom/notes/v0.11-front-list-filters-seo.md`
- `custom/notes/v0.21-postlaunch-screenshot-guide.md`

进入本轮前已经确认：
- `http://127.0.0.1:3000/` 可返回 200
- `http://localhost:3000/` 可返回 200
- `/admin/login` 可返回 200
- `/knowledge` 连接可以建立，但长时间无响应，最终超时

因此本轮不再排查端口、数据库服务是否启动这类基础问题，而是只聚焦：
- `/knowledge` 页面入口
- `getPublishedKnowledgeList`
- Prisma 查询
- 页面服务端渲染链路

## 3. 问题分析

表面现象是：
- `/knowledge` 请求连接成功
- 但 10 秒到 60 秒内没有收到任何响应字节
- `curl` 与 `Invoke-WebRequest` 都会超时

初看最像：
- Prisma 查询过慢
- 分类 / 标签 / 分页聚合过重
- 页面渲染中有 Promise 没有 resolve

但逐层验证后发现：
1. `app/(site)/knowledge/page.tsx` 页面入口本身很薄，只是读取 `searchParams` 后调用 `getPublishedKnowledgeList`
2. `generateMetadata` 与 `buildPageMetadata` 不包含远程请求，不像阻塞点
3. 手工将 `/knowledge` 的 4 段 Prisma 查询拆开后，全部都能在毫秒级到几百毫秒内返回
4. 将 `content.count / category.findMany / tag.findMany` 用 `Promise.all` 并发执行，也能在 643ms 内返回
5. 因此“Prisma 查询卡死”这个假设被否定

进一步对照后，最关键的发现是：
- 同样的代码在 `next start` 生产模式下，`/knowledge` 会立即返回 200，并完整输出 HTML
- 问题只出现在本机 `next dev` 路径

这说明真正的核心矛盾不是：
- `/knowledge` 的业务查询逻辑错误

而是：
- 当前本机的 `next dev` 运行器对前台列表页路径存在开发态异常，导致请求在 dev 模式下挂住

## 4. 候选方案比较

### 方案 A：继续深挖 Prisma 查询并重写 `/knowledge` 列表查询

核心思路：
- 继续砍掉分类、标签、分页等查询
- 把 `/knowledge` 改成极简列表

优点：
- 理论上可能绕开某些复杂查询风险

缺点：
- 已经被直接计时验证否定，查询本身不慢
- 会破坏已有前台结构
- 属于朝错误方向继续改业务代码

结论：
- 不采用

### 方案 B：最小修正页面细节，同时把本地开发运行器切换为更稳定的 webpack 模式

核心思路：
- 保持业务页面和数据底座不重写
- 仅做一处低风险页面细节修正
- 将 `npm run dev` 从 `next dev` 切为 `next dev --webpack`

优点：
- 改动最小
- 符合“修本地开发态，不推翻业务逻辑”的目标
- 与生产模式正常的事实相一致

缺点：
- 说明当前问题更偏运行器兼容性，而不是纯业务 bug
- 不能把锅完全归到 `/knowledge` 单页代码本身

结论：
- 本轮采用

## 5. 最终决策

本轮最终做了两个最小动作：

1. 对 `/knowledge` 列表页中的详情链接补一层 `encodeURIComponent(item.slug)`
   - 这是一个防御性修复
   - 因为当前已发布内容中存在中文 `slug`
   - 这样可以避免列表页直接输出原始中文路径

2. 将本地开发脚本从：
   - `next dev`
   改为：
   - `next dev --webpack`

这个决策的依据是：
- Prisma 查询并不慢
- 生产模式的 `/knowledge` 正常
- 问题主要出现在本机 dev 运行态

## 6. 具体实现

本轮修改的文件有：

1. `app/(site)/knowledge/page.tsx`
   - 将知识列表卡片跳转链接改为对 `slug` 做 URL 编码

2. `package.json`
   - 将 `dev` 脚本改为 `next dev --webpack`

3. `AGENTS.md`
   - 更新最新一期正式记录路径

## 7. 本轮优点

这轮方案的好处是：

1. 避免把问题误判成 Prisma 或 Schema 层问题
2. 保住了 `/knowledge` 页面的现有结构和业务表达
3. 让本地开发路径优先恢复可用，而不是继续大改列表页实现
4. 对中文 `slug` 做了更稳妥的 URL 编码处理

## 8. 本轮缺点与代价

这轮也有明确代价：

1. 根因更偏 Next.js 本地开发运行器，而不是单纯业务代码问题
2. `dev` 脚本从 Turbopack 切回 webpack，会牺牲一部分开发态启动体验
3. 这轮没有证明所有“前台列表页”问题都只靠页面级代码能解决，实际更像运行器兼容性问题

## 9. 验证与结果

本轮做了这些验证：

1. 单独计时 `/knowledge` 的 Prisma 查询
   - `content.count`：326ms
   - `category.findMany`：16ms
   - `tag.findMany`：8ms
   - `content.findMany`：9ms

2. 按页面真实写法并发执行查询
   - `Promise.all` 在 643ms 内返回

3. 检查实际已发布数据
   - 发现存在一条带中文 `slug` 的 `Content`

4. 用生产模式验证
   - `next start` 下：
     - `/knowledge` 返回 200
     - 首页返回 200
     - `/admin/login` 返回 200

5. 静态校验
   - `npm run typecheck` 通过
   - `npm run lint` 通过
   - `npm run build` 通过

可以确认的结论：
- `/knowledge` 的 Prisma 查询链路不是阻塞根因
- 当前问题主要出现在本机 `next dev` 运行态

## 10. 本轮结论

本轮最重要的结论是：

`/knowledge` 超时的主因并不是页面数据查询本身，而是当前本机 `next dev` 运行器在前台列表页路径上的开发态异常。

同时，本轮也确认了一个应默认继承的经验：
- 前台列表页中的中文 `slug` 应统一做 URL 编码，避免本地开发环境和不同运行器对路径处理不一致

## 11. 下一轮建议

下一步最值得做的 3 件事是：

1. 在真实浏览器里用当前 webpack dev 再验证一次：
   - `/knowledge`
   - `/terms`
   - `/brands`

2. 如果确认 webpack dev 稳定，就保留这次 `package.json` 的改动

3. 如果后续还想回到 Turbopack，需要单独做一次：
   - 前台列表页 dev 兼容性验证
   - 不要再把问题误判成 Prisma 慢查询
