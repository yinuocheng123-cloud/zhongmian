# 版本记录：v0.23-homepage-industry-entry-completion

## 1. 本轮目标

本轮目标是只调整首页布局与信息层级，在不改数据库结构、不扩展新功能、不影响后台的前提下，把中眠网首页收口成“不缺不漏”的行业入口首页。

这轮优先级属于：

- 功能补充
- 结构重构
- 可维护性提升

核心要求不是“首页更好看”，而是让用户一眼看懂：

1. 中眠网是什么
2. 可以查睡眠知识
3. 可以看睡眠词库
4. 可以看睡眠品牌
5. 可以看行业趋势 / 行业事件 / 品牌进展
6. 可以理解平台有信任体系入口
7. 可以看见企业合作与品牌入驻入口

## 2. 背景与上下文

本轮承接以下记录：

- v0.4-industry-entry-restructure
- v0.5-home-entry-intensity
- v0.6-visual-tone-upgrade
- v0.22-knowledge-dev-timeout

修改前首页已经有较强的“行业入口”表达，但还存在两个问题：

1. 首页更多是在表达“入口逻辑已经成立”，但还不够贴合本轮要求的七大入口命名与顺序。
2. 首页的词库、知识、品牌等中段区块在缺少真实已发布数据时，容易变成结构正确但信息偏空的状态，不利于交付时直接检查。

同时，当前项目前台三条主线已经有真实 Prisma 列表链路，因此首页不应再完全依赖 demo 数据表达，否则会和现有系统状态脱节。

## 3. 问题分析

表面现象：

- 首页结构并不混乱，但仍有“像门户 demo，不像正式行业入口首页”的风险。
- 中段内容区如果真实数据为空，会让“词库精选 / 知识内容 / 品牌入口”看起来像空壳。

核心矛盾：

- 首页既要保留“行业入口优先”的明确结构，又要避免在真实数据尚少时显得内容缺口过大。
- 如果完全改成纯真实数据渲染，当前数据库为空时，首页会丢失入口厚度。
- 如果继续完全依赖 demo 数据，首页又会和当前真实前台链路脱节。

主要原因：

1. 首页原实现更多依赖 `lib/demo-data.ts` 中的静态结构。
2. 当前真实 Prisma 数据在本地环境并不稳定充足，不能保证首页中段永远有可展示内容。
3. 用户本轮只允许改首页，不允许改后台和三条主线逻辑，因此首页只能在自身做最小平衡。

## 4. 候选方案比较

### 方案 A：继续完全使用 demo 数据

核心思路：

- 保持首页纯静态演示表达，只重排区块和文案。

优点：

- 改动最小
- 不依赖数据库状态

缺点：

- 首页会和当前真实前台链路脱节
- “词库精选 / 知识内容 / 品牌入口”看起来不像真实系统
- 不利于后续继续往真实运营首页演进

放弃原因：

- 不符合当前系统已经具备真实数据链路的阶段状态

### 方案 B：首页完全切换为真实 Prisma 数据

核心思路：

- 首页所有中段区块都直接依赖真实 `Content / Term / Brand` 数据

优点：

- 和当前系统一致
- 首页更接近正式线上站点

缺点：

- 当前无已发布数据时首页会显得偏空
- 不满足“不缺不漏”的交付要求

放弃原因：

- 当前阶段首页不能依赖数据库内容完备度

### 方案 C：入口结构静态显性 + 中段区块真实数据优先 + 结构化兜底

核心思路：

- 首屏、七大入口、信任体系、合作入口等结构维持显性静态表达
- 词库精选、知识内容、品牌入口优先读取真实 Prisma 数据
- 若真实数据为空，则用结构化精选卡片兜底，而不是显示空白或弱提示

优点：

- 同时满足入口表达、真实链路对齐和交付可看性
- 不改数据库结构
- 不改三条主线页面逻辑

缺点：

- 首页同时存在“真实数据渲染”和“结构化兜底”两套来源
- 后续如果要完全线上化，仍应把兜底卡逐步让位给后台配置

最终选择原因：

- 这是本轮“最小改动、最大收益”的方案

## 5. 最终决策

最终采用方案 C。

关键设计决策：

1. 首页改为显性的九段结构：
   - 首屏平台定位区
   - 七大核心入口区
   - 热门问题入口区
   - 睡眠词库精选区
   - 知识内容 / 行业趋势区
   - 品牌入口区
   - 信任体系区
   - 企业合作 / 品牌入驻区
   - 最新内容区
2. 首页直接调用现有 `getPublishedKnowledgeList / getPublishedTermsList / getPublishedBrandsList`
3. 当真实数据为空时，不显示空白，而是显示结构化精选卡，保持首页“入口完整感”
4. 最新内容继续存在，但放到最后，确保首页不被内容流主导
5. 后台入口只做轻量露出，放在首屏辅助卡中，不把前台做成后台跳转页

## 6. 具体实现

改动文件：

### 1. `app/(site)/page.tsx`

作用：

- 把首页从静态 demo 入口页重组为行业入口首页
- 接入现有真实列表数据
- 在无真实数据时使用术语 / 知识 / 品牌结构卡兜底

具体改动：

- 将首页改为异步服务端组件
- 引入：
  - `getPublishedKnowledgeList`
  - `getPublishedTermsList`
  - `getPublishedBrandsList`
- 删除对首页原 demo 结构数组的直接依赖
- 按本轮要求重组首页九个模块
- 增加 slug 编码辅助，避免首页详情链接在中文 slug 情况下有潜在风险

### 2. `custom/notes/v0.23-homepage-industry-entry-completion.md`

作用：

- 记录本轮首页结构调整的背景、取舍、实现和后续建议

### 3. `AGENTS.md`

作用：

- 将“当前最新一期正式记录”更新到 v0.23，保证后续任务能优先读到这轮结论

## 7. 本轮优点

当前方案相比之前的好处：

1. 首页显性回答了“中眠网是什么”
2. 首页显性回答了“知识入口、词库入口、品牌入口在哪里”
3. 首页显性补齐了“行业趋势 / 行业事件 / 品牌进展 / 中眠智库 / 行业参考”
4. 首页显性补齐了“企业合作 / 品牌入驻”
5. 内容流被稳定压到后半段，首页不再容易退回普通资讯站表达
6. 即使数据库内容不足，首页中段也不会显得空白
7. 没有改动后台、数据库或三条主线页面逻辑，改动边界干净

## 8. 本轮缺点与代价

这一轮仍有已知代价：

1. 首页的词库 / 知识 / 品牌中段区块，目前是“真实数据优先 + 结构兜底”双轨模式
2. 如果以后需要完全后台可配置首页，这套兜底结构可能还要继续演进
3. 行业趋势 / 行业事件 / 品牌进展当前仍是入口语义和区块表达，不是独立数据模型
4. 后台入口虽然已经做轻量露出，但后续如果客户希望更克制，仍可能需要再压一层

这些都不是本轮阻断项，但属于后续迭代空间。

- 这不是本轮首页改动带来的新问题，而是当前本地数据库服务状态问题

## 10. 本轮结论

本轮最重要的结论有三个：

1. 首页可以只通过一个页面改动，就把“基础知识入口 + 信任入口 + 商业入口”的结构压实。
2. 首页不应该在当前阶段完全依赖真实数据，也不应该完全依赖 demo 数据，混合策略更适合交付阶段。
3. 当前首页已经更接近“行业入口首页”，而不是“内容门户首页”。

## 11. 下一轮建议

下一步最值得继续做的 1~3 件事：

1. 若用户认可当前首页方向，可做一次小范围 commit 收口
2. 若本地数据库恢复稳定，可再看首页中段是否要进一步提高真实数据占比
3. 若后续进入正式上线准备，可把首页部分入口和精选卡位逐步转为后台可配置
