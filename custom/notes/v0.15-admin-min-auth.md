# 版本记录：v0.15-admin-min-auth

## 1. 本轮目标

本轮目标是为中眠网后台补齐“最小可用鉴权”，让 `/admin` 不再只是可直接进入的演示骨架，而是真正具备后台访问控制能力。

本轮优先解决的问题是：
- 后台主面板、内容管理、词条管理、品牌管理、AI 编辑部当前都可被未登录用户直接访问
- 登录页此前只是 Demo 展示页，没有真实登录能力
- 上线前如果后台没有最小鉴权，会直接阻断正式商用

本轮属于：
- 功能补充
- 上线阻断修复
- 可运维性补强

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.14-provider-layer.md`

进入本轮之前，项目已经具备：
- 前台公开站点主路径
- Prisma 数据底座与后台真实录入
- 发布流转
- AI 编辑部基础工作流

但后台访问控制仍停留在“页面骨架已在、鉴权尚未接入”的状态。这意味着系统虽然能工作，但后台不具备正式可用条件。

## 3. 问题分析

表面现象：
- `/admin/login` 只是展示说明，没有真实登录
- `/admin` 及其子路由缺少最小访问门槛

核心矛盾：
- 当前最需要的是“先可用、先挡住未登录访问”，而不是一步到位做复杂账号体系

真正需要解决的点有三个：
1. 建立一个最小但真实可用的登录态
2. 让后台页面和后台数据操作都能识别登录态
3. 不影响现有前台公开访问逻辑

## 4. 候选方案比较

### 方案 A：直接接入完整鉴权库或第三方登录

核心思路：
- 引入成熟鉴权框架，建立完整用户模型、角色权限、登录流程

优点：
- 长期能力更完整
- 后续扩展 RBAC 更自然

缺点：
- 改动面大
- 需要新增依赖、账号模型甚至迁移
- 不符合本轮“只补最小可用鉴权”的要求

结论：
- 本轮不采用

### 方案 B：环境变量账号密码 + 服务端签名 cookie

核心思路：
- 后台账号和密码先使用环境变量配置
- 登录后由服务端签名写入 httpOnly cookie
- 后台页面与后台服务层统一校验 cookie

优点：
- 改动小
- 不需要新增依赖
- 能快速满足正式商用前的最小后台访问控制要求

缺点：
- 只有单账号级能力
- 暂不支持角色权限与多用户管理
- 账号轮换依赖环境变量变更

结论：
- 本轮采用

## 5. 最终决策

本轮最终采用“环境变量账号密码 + 服务端签名 session cookie + 后台页面与服务层双重守卫”的方案。

关键决策如下：
1. 账号密码不入库，先使用 `ADMIN_USERNAME`、`ADMIN_PASSWORD`、`ADMIN_SESSION_SECRET`
2. 开发环境允许回退到默认账号，避免本地调试被环境变量阻塞
3. 登录态使用服务端签名 cookie，避免把敏感信息暴露给前端脚本
4. 后台页面入口和后台 Prisma 读取/写入服务都进行登录校验，避免只挡页面不挡动作
5. 登录与登出改为标准 POST 路由，优先保证兼容性和可验证性

## 6. 具体实现

### 6.1 会话层

修改文件：
- `features/admin/auth/session.ts`

作用：
- 统一管理后台鉴权配置
- 生成和校验签名 cookie
- 提供 `requireAdminSession`、`createAdminSession`、`clearAdminSession`

### 6.2 登录页改为真实可用

修改文件：
- `app/admin/login/page.tsx`

作用：
- 登录页不再只是 Demo
- 使用标准表单提交到登录路由
- 可展示登录失败、登出成功、配置状态提示
- 已登录用户访问登录页时自动跳转到 `/admin`

### 6.3 登录与登出提交入口

新增文件：
- `app/admin/login/submit/route.ts`
- `app/admin/logout/route.ts`

作用：
- 负责接收登录表单、校验账号密码、写入/清除 session cookie
- 通过标准 HTTP POST 流程替代之前不稳定的提交方式

### 6.4 后台布局与登出入口

修改文件：
- `components/admin/admin-shell.tsx`
- `app/admin/(panel)/layout.tsx`

作用：
- 所有后台面板统一先过登录校验
- 侧栏显示当前登录账号
- 登出入口切换为真实 POST 提交

### 6.5 后台数据层守卫

修改文件：
- `features/admin/resources/server.ts`
- `features/admin/resources/actions.ts`
- `features/admin/editorial/server.ts`
- `features/admin/editorial/actions.ts`

作用：
- 后台列表读取、编辑读取、保存动作在服务层再次校验登录态
- 把后台操作人从原来的 demo 标记替换为统一的真实后台操作者名

## 7. 本轮优点

本轮完成后，后台最小可用条件已经具备：
- 未登录用户无法直接进入后台核心页面
- 登录页可真实使用
- 登录成功后可访问后台页面与后台数据操作
- 登出后登录态立即失效
- 前台公开访问逻辑未受影响

从上线角度看，本轮最大的价值是把“后台仍是演示态”的阻断问题清掉了。

## 8. 本轮缺点与代价

这套方案有明确边界：
- 仍然只有最小单账号能力
- 暂不支持多账号、角色权限、账号锁定、密码重置
- 深层后台页面被拦截后，当前默认统一回到 `/admin/login?next=/admin`，没有精确保留原目标页
- 还有旧的登录表单客户端组件和旧 server action 文件保留在仓库中，但本轮没有继续清理，目的是保持改动最小

这些都不影响当前“后台正式可用”的目标，但如果未来进入多人协作后台，就需要升级鉴权体系。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`
4. 未登录访问以下页面，确认会被重定向到登录页：
   - `/admin`
   - `/admin/content`
   - `/admin/terms`
   - `/admin/brands`
   - `/admin/ai-editorial`
5. 使用开发环境默认账号完成登录，确认：
   - 登录后会写入 `zhongmian_admin_session`
   - 可访问 `/admin/content`
6. 执行登出，确认：
   - session cookie 被清除
   - 再访问 `/admin` 会重新跳回登录页

验证结果：
- 全部通过

## 10. 本轮结论

本轮最重要的结论是：
中眠网后台已经从“可直接访问的演示后台”，切换为“具备最小真实访问控制的后台”。

被证实的判断有：
- 现阶段不必引入重型鉴权库，也能先满足正式商用前的后台访问门槛
- 服务端签名 cookie 足以支撑当前单后台账号场景
- 只要页面入口和数据服务层都做校验，就能在最小改动下建立较可靠的后台访问控制

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 上线前把生产环境后台账号与 session secret 配齐
   - 收益最高
   - 这是正式商用前的实际部署条件

2. 做一轮后台人工回归
   - 验证真实浏览器中的登录、内容保存、登出链路
   - 风险最低、收益直接

3. 如果后续进入多人协作后台，再评估升级到多账号 / RBAC
   - 当前不是阻断项
   - 适合放到正式上线后的下一阶段
