# 版本记录：v0.11-front-list-filters-seo

## 1. 本轮目标

本轮目标是在前台主干路径已经打通的基础上，继续完成三件事：

1. 把 `/knowledge`、`/terms`、`/brands` 三个栏目页从“能读真实数据”推进到“可筛选、可分页、可搜索”的最小可用状态。
2. 优化 `Content / Term / Brand` 详情页之间的关联逻辑，让相关推荐、相关词条、相关品牌不再只是弱占位。
3. 补齐 SEO 基础能力，包括统一 metadata 结构、列表页索引策略与 `sitemap.xml`。

这一轮属于：
- 功能补充
- 结构强化
- SEO 基础建设

## 2. 背景与上下文

本轮承接以下记录：
- `v0.10-site-list-pages-prisma`
- `v0.9-site-detail-pages-prisma`
- `v0.8-admin-crud-workflow`

开始本轮时，系统已经具备：
- 后台真实录入、保存、切状态与发布流转
- 前台详情页读取 Prisma 数据
- 前台列表页读取 Prisma 数据

但仍存在三个明显缺口：

1. 列表页虽然已经接上 Prisma，但还缺少真正可用的筛选、分页与更完整的搜索入口。
2. 详情页的关联内容仍然偏“最小占位”，没有充分利用分类、标签和跨实体关系。
3. 前台公开路径虽然成立，但 SEO 结构不够完整，metadata 与 sitemap 还没有形成统一口径。

## 3. 问题分析

表面现象是：
- 栏目页已经不是 demo 数据了
- 详情页也已经能展示真实内容

但真正的问题是：
- 用户进入列表页后，缺少进一步缩小范围的结构化入口
- 详情页之间虽然已经有“相关推荐”字样，但关联依据还比较薄
- 搜索引擎虽然能访问页面，但页面 title / description / canonical / sitemap 没有统一体系

核心矛盾不是“页面能不能打开”，而是：
- 中眠网是否已经从“真实可看”进入“真实可运营、可被索引、可被持续扩展”的阶段

## 4. 候选方案比较

### 方案 A：只补列表页筛选，不动详情关联与 SEO

核心思路：
- 先把分类筛选、标签筛选和分页加到栏目页里
- 其余问题继续留到下一轮

优点：
- 改动集中
- 风险较低

缺点：
- 详情页关联能力仍然偏弱
- SEO 依然处于半成品状态
- 用户访问路径虽然更顺，但搜索入口质量提升有限

结论：
- 不采用。因为这会让前台访问路径继续处于“功能已通、站点能力未成型”的中间状态。

### 方案 B：按阶段连续推进“筛选分页 + 关联优化 + SEO 基础”

核心思路：
- 先把列表页结构能力补齐
- 再增强详情页关联查询
- 最后补上 metadata 与 sitemap

优点：
- 三件事共享同一批公开查询基础
- 一轮完成后，前台公开路径会明显更完整
- 后续继续接筛选聚合或 AI 编辑部时不用回头返工

缺点：
- 横跨文件更多
- 需要更严格地做类型和构建验证

结论：
- 本轮采用。

## 5. 最终决策

本轮最终采用“先补栏目页能力，再补详情页关联，最后补 SEO 基础层”的连续推进方案。

关键决策如下：

1. 列表页查询层统一支持：
   - `q`
   - `category`
   - `tag`
   - `page`
   - 固定 `take`
2. 列表页统一只读取 `PUBLISHED` 数据，继续与详情页公开逻辑保持一致。
3. 详情页关联优化优先复用已有分类、标签和跨实体关系，不新增数据库表，不推翻 Schema。
4. 列表页的搜索、筛选与分页页统一 `noindex`，避免 query 参数页重复收录。
5. `sitemap` 只输出公开可访问路径和已发布详情页，不输出后台路径。

## 6. 具体实现

### 6.1 列表页查询层升级

修改文件：
- `features/site/lists/server.ts`

本轮新增能力：
- 分类筛选
- 标签筛选
- 分页
- 最小搜索
- 分类计数
- 标签计数

三类查询分别为：
- `getPublishedKnowledgeList`
- `getPublishedTermsList`
- `getPublishedBrandsList`

所有查询统一具备：
- `workflowStatus = PUBLISHED`
- `q / category / tag / page / take`
- `pagination.total / totalPages / pageSize`
- 数据库未就绪时明确返回“真实列表未就绪”

### 6.2 前台列表页重构为真实筛选入口

修改文件：
- `app/(site)/knowledge/page.tsx`
- `app/(site)/terms/page.tsx`
- `app/(site)/brands/page.tsx`
- `components/site/list-controls.tsx`

具体结果：

#### `/knowledge`
- 仍然保持“问题入口 + 知识入口”角色
- 接入真实分类筛选
- 接入真实标签筛选
- 接入分页
- 接入最小搜索
- 保持知识卡片，而不是普通资讯流

#### `/terms`
- 明确为术语目录 / 概念入口 / 知识节点
- 接入真实分类筛选
- 接入真实标签筛选
- 接入分页
- 接入最小搜索
- 卡片突出定义而不是文章属性

#### `/brands`
- 明确为行业名录 + 信任展示 + 商业入口
- 接入真实分类筛选
- 接入真实标签筛选
- 接入分页
- 接入最小搜索
- 卡片以企业名片结构为主

### 6.3 详情页关联能力优化

修改文件：
- `features/site/details/server.ts`

#### Content 详情关联优化
- 相关推荐不再只看单一条件
- 同时考虑：
  - 分类重合
  - 标签重合
  - 相关词条
  - 相关品牌

#### Term 详情关联优化
- 相关词条不再只做弱占位
- 同时考虑：
  - 分类重合
  - 标签重合
  - 关联内容重合

#### Brand 详情关联优化
- 相关品牌综合考虑：
  - 地区
  - 分类
  - 标签
- 相关内容综合考虑：
  - 直接关联品牌
  - 品牌标签重合
  - 相关品牌反向带出的内容

本轮没有新建关系表，优先用已有 Schema 把关联能力做厚。

### 6.4 SEO 基础层建设

新增文件：
- `lib/seo.ts`
- `app/sitemap.ts`

修改文件：
- `app/layout.tsx`
- `app/(site)/page.tsx`
- `app/(site)/about/page.tsx`
- `app/(site)/knowledge/page.tsx`
- `app/(site)/terms/page.tsx`
- `app/(site)/brands/page.tsx`

本轮新增能力：

1. 统一站点基础信息
   - `SITE_NAME`
   - `SITE_URL`
   - `SITE_DESCRIPTION`

2. 统一页面 metadata 构建
   - canonical URL
   - title / description
   - open graph 基础字段
   - `noindex` 控制

3. 列表页索引策略
   - 主路径可索引
   - 搜索页、筛选页、分页页 `noindex`

4. `sitemap.xml`
   - 输出首页、栏目页、关于页
   - 输出所有已发布的：
     - `/knowledge/[slug]`
     - `/terms/[slug]`
     - `/brands/[slug]`
   - 数据库未配置时仅输出静态公开路径

## 7. 本轮优点

相比 `v0.10`，本轮最直接的提升有：

1. 前台三条主线都已经具备最小可用的筛选、分页与搜索结构。
2. 列表页不再只是“能显示内容”，而是开始具备真正的目录与入口能力。
3. 详情页之间的关联更合理，前台内容消费不再那么孤立。
4. 站点已经具备统一 metadata 与 `sitemap.xml`，SEO 基础层正式成立。
5. 整个前台公开路径从“已打通”推进到了“可运营、可索引”的阶段。

## 8. 本轮缺点与代价

这一轮也有明确代价和未完成项：

1. 列表页搜索仍然是最小实现，没有做高级检索和多条件组合面板。
2. 分页目前是常规页码分页，没有做 SEO 更细的分页策略或更强的结果聚合。
3. 详情页关联仍然基于现有关系和启发式规则，没有独立的推荐模型。
4. 详情页 metadata 仍然沿用上一轮的实现口径，虽然功能已可用，但与本轮 `lib/seo.ts` 的统一封装还没有完全合并。
5. 当前本地没有 `DATABASE_URL`，因此运行时仍然只能看到“真实数据未就绪”兜底，无法在本机完成真实数据库联调。

这些代价是本轮可接受的，因为本轮目标是先把前台公开路径能力做完整，而不是一次做完高级搜索和完整 SEO 体系。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run prisma:generate`
2. `npm run typecheck`
3. `npm run lint`
4. `npm run build`

验证结果：
- 全部通过
- 构建输出中可见：
  - `/knowledge`

  - `/brands`
  - `/knowledge/[slug]`
  - `/terms/[slug]`

本轮中途曾出现一次类型错误：
- `features/site/details/server.ts` 中品牌关联内容查询的 `where` 条件数组被 TypeScript 推窄
- 最终通过显式声明 `Prisma.ContentWhereInput[] / Prisma.BrandWhereInput[] / Prisma.TermWhereInput[]` 解决

## 10. 本轮结论

本轮最重要的结论是：

中眠网前台已经不只是“后台可发、前台可看”，而是开始具备真实站点的目录能力、关联能力和 SEO 基础能力。

本轮被证实的判断有：
- 统一查询层非常适合继续承接前台筛选、聚合和 SEO 输出
- 不新增表结构，也可以先把详情页关联能力做出明显提升
- 列表页搜索/筛选页统一 `noindex` 是当前阶段最简单稳妥的方案

本轮应默认继承的经验有：
- 公开前台始终只消费 `PUBLISHED`
- 数据库未就绪时宁可明确兜底，也不回退 demo 数据制造假象
- 中眠网的栏目页必须保持各自角色，不要因为接了真实数据就长成普通内容列表

## 11. 下一轮建议

下一轮最值得继续推进的方向有三个：

1. 前台列表页进一步筛选与聚合优化
   - 收益最高
   - 现在的查询层已经具备继续扩展的基础

2. SEO / GEO 进一步完善
   - 可继续统一详情页 metadata
   - 可继续补 robots、结构化数据、栏目级 SEO 文案

3. AI 编辑部基础工作流
   - 现在前后台主路径已基本成立，已具备把 `AiTask` 进一步接到真实内容流转的条件

如果只选一个，我会优先推荐：
- 前台列表页进一步筛选与聚合优化

因为当前前台主路径已经成立，继续增强栏目页的结构化聚合能力，能最直接提升中眠网“行业入口”的使用价值。
