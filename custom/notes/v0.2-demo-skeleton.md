# 版本记录：v0.2-中眠网项目骨架与 demo 落地

## 1. 本轮目标

本轮目标是在 `v0.1` 技术方案基线确认后，将方案推进到“可确认方向的工程骨架 demo”阶段。

本轮重点不是实现完整业务功能，而是先落地：

- Git 未绑定情况下的最小建仓步骤说明
- Next.js + Tailwind + Prisma + PostgreSQL 的工程骨架
- 前台首页、睡眠词库、睡眠知识、睡眠品牌四个 demo 页面
- 后台登录页、控制台、内容管理、AI 编辑部的骨架页面
- 可扩展的路由结构、布局结构和目录结构

本轮属于：

- 功能补充前的骨架落地
- 可维护性提升
- 正式开发前的方向确认

## 3. 问题分析

### 3.1 本轮要解决的核心问题

用户希望先确认产品结构和门户感觉是否正确，而不是立刻进入数据库、权限和后台表单细节。

因此本轮真正要解决的问题是：

- 如何让首页一眼看出“这是垂直行业门户”，而不是普通资讯站模板
- 如何让前台三条主线与后台三条主线形成映射关系
- 如何在不做重业务逻辑的前提下，把扩展方向预埋清楚
- 如何让工程骨架能稳定 build，而不是只停留在静态示意图

### 3.2 需要避免的问题

本轮明确不做：

- Prisma Schema 正式建模
- 权限系统真实实现
- 后台表单与发布流细节
- AI 编辑部的提示词与模型调用细节
- 词条、品牌、知识详情页与搜索系统

## 4. 候选方案比较

### 方案 A：只做静态页面，不接工程依赖

**核心思路**

只搭几张页面，不安装 Prisma，也不整理目录结构。

**优点**

- 速度最快
- 视觉确认成本低

**缺点**

- 不能代表真实开发骨架
- 下一轮仍要返工项目初始化
- 无法验证布局结构与路由组织是否合理

**结论**

不选。用户明确要求“项目骨架”和“可开发版本”。

### 方案 B：直接进入 Prisma Schema、权限、后台表单

**核心思路**

骨架一搭好就开始真实业务开发。

**优点**

- 进度表面上更快

**缺点**

- 在视觉结构和产品方向尚未确认前，过早进入细节开发
- 与用户本轮目标不一致

**结论**

不选。本轮必须先做方向确认型 demo。

### 方案 C：先完成可运行的门户 demo + 后台骨架

**核心思路**

工程初始化到位，前后台骨架可运行、可 build，但数据与流程只做占位。

**优点**

- 既能确认产品结构，也能保留真实工程上下文
- 下一轮可以直接接 Prisma Schema 和后台功能
- 风险和返工成本最低

**缺点**

- 需要在 demo 和正式代码之间把边界控制清楚

**结论**

最终采用本方案。

## 5. 最终决策

### 5.1 前台决策

前台采用：

- `app/(site)` route group 组织门户页面
- 首页突出高信息密度的门户结构
- 睡眠词库、睡眠知识、睡眠品牌三页先做频道骨架
- 关于我们页面补齐站点基本说明

### 5.2 后台决策

后台采用：

- `app/admin/login` 作为独立登录入口
- `app/admin/(panel)` route group 承载登录后的后台布局
- 控制台、内容管理、AI 编辑部先做骨架页
- 词条、品牌、SEO 等模块先保留菜单位置，不提前落业务

### 5.3 数据层决策

数据层采用：

- 接入 `prisma` 与 `@prisma/client`
- 建立 `prisma/schema.prisma` 占位文件
- 建立 `lib/prisma.ts` 单例入口
- 使用 `.env.example` 而不是直接生成真实 `.env`

### 5.4 版本策略决策

考虑到 Prisma 7 的配置变更会在骨架阶段引入额外噪音，本轮将 Prisma 降级到稳定版本 `6.17.1`，优先保证骨架可运行和后续开发顺畅。

## 10. 本轮结论

本轮最重要的结论是：

中眠网的方向可以通过“门户首页 + 三个核心频道 + 后台双主线”清晰表达出来，而且这个方向已经落成了可 build 的代码骨架。

本轮确认了以下判断：

- 首页高信息密度路线是成立的
- 词库、知识、品牌三条主线能自然构成一期前台结构
- 后台同时展示 CMS 和 AI 编辑部是合理的
- 先确认骨架，再进入正式建模和流程细化，是更稳妥的推进方式

下一步最值得做的事情建议如下：

3. 然后再进入后台真实表单、发布流转和 AI 编辑部基础流程的正式开发
