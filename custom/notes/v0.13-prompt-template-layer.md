# 版本记录：v0.13-prompt-template-layer

## 1. 本轮目标

本轮目标是在不接入真实 AI 模型、不调整既有 Schema 的前提下，为中眠网 AI 编辑部建立“提示词模板层 + 结构化输入层”。

这件事值得优先处理的原因是：

- `AiTask` 已经接入内容生产流程，但生成前输入仍然偏散，后续真接模型时容易把表单、动作和生成逻辑重新拆一遍。
- 现有“占位生成”可以证明流程成立，但还不能证明未来能稳定替换成真实模型。
- 模板层一旦做对，后续接真实模型时只需要替换生成器，而不是重写任务系统。

本轮属于：

- 功能补充
- 结构补强
- 可扩展性提升

## 2. 背景与上下文

本轮承接以下记录：

- `v0.12-ai-editorial-workflow`
- `v0.11-front-list-filters-seo`

进入本轮之前，系统已经具备：

- `Content / Term / Brand` 的统一内容底座
- 后台真实录入与发布流转
- 前台列表与详情的真实数据展示
- AI 编辑部最小工作流：选题 -> 占位生成 -> Content 草稿挂接

但 AI 编辑部仍有明显短板：

- 任务输入结构主要是自由字段，没有明确模板边界
- 不同任务类型之间没有统一输入约束
- 占位生成只是字符串拼接，未来替换真实模型时没有稳定接口
- 历史任务与新结构之间缺少兼容策略

## 3. 问题分析

表面问题是“AI 任务页面缺少模板选择和结构化字段”，但更核心的矛盾是：

- 如果模板规则写死在动作层，那么所谓模板层只是换个名字的硬编码
- 如果只做新模板，不处理旧 `inputPayload`，历史任务一旦编辑就会出现兼容断裂
- 如果占位生成仍然不产出统一结构，未来接真实模型时输出格式仍会飘

所以本轮真正要解决的不是“多几个表单项”，而是三层关系：

1. 模板如何独立于代码存在
2. 任务输入如何被标准化
3. 模拟生成如何稳定输出结构化结果

## 4. 候选方案比较

### 方案 A：继续沿用自由输入，只在表单上增加几个字段

核心思路：

- 不引入模板定义
- 只在现有表单上补充关键词、目标等字段

优点：

- 实现成本最低
- 对现有代码改动最少

缺点：

- 模板规则依然散落在动作层
- 无法支持不同实体或不同内容目标的输入约束
- 后续接模型时仍然要重做一遍结构层

结论：

- 不采用。它只能让页面更像 AI 表单，不能让系统具备真正的模板能力。

### 方案 B：建立外置 JSON 模板 + 结构化输入 + 模拟统一输出

核心思路：

- 模板定义放到 `custom/ai-editorial/templates/`
- 服务层统一读取模板
- 表单围绕模板驱动输入选项
- 动作层保存结构化 `inputPayload`
- 占位生成按模板产出统一文本与结构化结果

优点：

- 模板不再写死在业务代码中
- 输入和输出都被收敛成统一结构
- 后续接真实模型时替换点清晰
- 可以兼容旧版任务数据

缺点：

- 需要同时改动模板读取、表单、服务层、动作层
- 需要补一层历史任务兼容逻辑

结论：

- 本轮采用。

## 5. 最终决策

本轮最终采用“外置模板文件 + 结构化输入载荷 + 模板驱动占位生成”的方案。

关键设计决策如下：

1. 模板不写死在代码中，而是放在 `custom/ai-editorial/templates/` 下的 JSON 文件中。
2. `AiTask.inputPayload` 升级为 `schemaVersion: "v2"` 的结构化对象，包含：
   - `templateId`
   - `templateName`
   - `targetKind`
   - `input`
3. `input` 统一收敛为：
   - `title`
   - `topic`
   - `keywords`
   - `contentType`
   - `generationGoal`
   - `notes`
   - `shouldCreateDraft`
4. 保留对旧版载荷的兼容解析，不要求迁移数据库即可继续编辑历史任务。
5. 模拟生成不追求效果，只追求统一格式，因此同时产出：
   - `outputText`
   - `outputJson`
6. Content 挂接逻辑只在 `targetKind === "CONTENT"` 时生效，避免在当前阶段误做 Term / Brand 草稿逻辑。

## 6. 具体实现

本轮新增 / 修改如下：

### 6.1 新增模板文件目录

新增：

- `custom/ai-editorial/templates/content-knowledge.json`
- `custom/ai-editorial/templates/content-topic.json`
- `custom/ai-editorial/templates/term-definition.json`

作用：

- 把知识科普、专题内容、词条定义三类模板以外置数据形式保存
- 每个模板统一定义：
  - `targetKind`
  - `defaultTaskType`
  - `supportedTaskTypes`
  - `supportedContentTypes`
  - `generationGoals`
  - `promptRules`
  - `outputSections`

### 6.2 新增模板服务层

新增：

- `features/admin/editorial/templates.ts`

作用：

- 从 `custom/ai-editorial/templates/` 读取 JSON 模板
- 统一校验模板结构
- 输出模板选项供页面复用
- 根据模板与输入生成：
  - 任务提示词文本
  - 结构化占位输出

### 6.3 升级类型与兼容工具

修改：

- `features/admin/editorial/types.ts`
- `features/admin/editorial/utils.ts`

作用：

- 新增模板、结构化输入、标准化载荷等类型
- 为旧版 `direction / desiredContentType` 载荷提供兼容解析
- 新增关键词解析与格式化工具

### 6.4 升级服务层查询

修改：

- `features/admin/editorial/server.ts`

作用：

- 新建页读取模板选项和内容候选项
- 编辑页把历史任务映射为临时模板选项，保证兼容编辑
- 列表页新增模板名称、目标实体展示

### 6.5 升级动作层

修改：

- `features/admin/editorial/actions.ts`

作用：

- 保存任务时按模板校验任务类型、内容类型、生成目标
- 写入结构化 `inputPayload`
- 生成模板驱动的 `prompt`
- 占位生成时写入统一的 `outputText / outputJson`
- 继续沿用既有 Content 草稿挂接、Workflow、ContentVersion 链路

### 6.6 升级后台页面与表单

修改：

- `components/admin/ai-task-form.tsx`
- `app/admin/(panel)/ai-editorial/page.tsx`
- `app/admin/(panel)/ai-editorial/new/page.tsx`
- `app/admin/(panel)/ai-editorial/[id]/page.tsx`

作用：

- 后台表单支持模板选择
- 输入结构改为标题、主题、关键词、内容类型、生成目标、备注
- 根据模板收敛可选项
- 列表页与详情页展示模板信息与目标实体

## 7. 本轮优点

相比上一轮，本轮有几个明确提升：

1. AI 任务不再只是“自由输入 + 字符串占位”，而是进入结构化输入阶段。
2. 模板来源清晰，已经从代码逻辑中抽离到可维护文件。
3. 模拟生成已有稳定替换点，未来接真实模型可以直接复用输入输出边界。
4. 保住了现有 `AiTask + Content + Workflow` 路径，没有推翻已有工作流。
5. 历史任务没有被放弃，具备兼容编辑能力。

## 8. 本轮缺点与代价

这轮仍然有明确边界：

1. 没有接真实模型，模板层当前仍然只服务于模拟生成。
2. 模板文件目前仍然是代码仓内静态 JSON，尚未进入后台可配置状态。
3. `Term / Brand` 模板虽已进入结构，但暂未接实体草稿创建逻辑。
4. 搜索与筛选还没有围绕模板维度单独展开，目前仍是最小可用列表。
5. 占位稿质量不是目标，当前只保证结构稳定，不保证内容完成度。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`

验证结果：

- 全部通过
- 构建产物中 `/admin/ai-editorial`、`/admin/ai-editorial/new`、`/admin/ai-editorial/[id]` 正常生成

本轮没有执行真实数据库联调，原因是当前验证目标是模板层与结构层是否闭合，而不是 AI 模型效果或生产数据写入效果。

## 10. 本轮结论

本轮最重要的结论是：

AI 编辑部已经从“可挂接内容流的任务系统”，进一步推进成“具备结构化输入与模板边界的可扩展生产系统”。

被证实的判断包括：

- 模板文件外置不会破坏现有工作流，反而让动作层更清晰
- 旧载荷兼容完全可以在服务层 / 工具层解决，不需要先做迁移
- 在不接真实模型的前提下，也能把未来的生成接口边界先立住

以后默认应继承的经验是：

- AI 能力优先做“结构化输入与流程接入”，不要先做效果幻觉
- 模板规则应尽量数据化，而不是散落在页面和 action 里

## 11. 下一轮建议

下一步最值得进入的方向有三个：

1. 提示词 / 模板层继续补强
   - 增加模板预览、模板说明区块、输出结构预览
   - 收益最高

2. 后台能力补强
   - 为 AI 任务增加模板筛选、目标实体筛选、结果摘要展示
   - 风险最低

3. 真实模型接入前的接口封装
   - 把当前 placeholder generator 再抽一层 provider 接口
   - 对未来接模型最直接

如果只选一个，我会优先推荐：

- 真实模型接入前的 provider 接口层

因为模板和结构化输入已经成立，下一步最自然的工作就是把“模拟生成器”和“未来真实模型生成器”的边界再隔离清楚。
