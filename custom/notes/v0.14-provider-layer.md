# 版本记录：v0.14-provider-layer

## 1. 本轮目标

本轮目标是在不接入真实 AI 模型、不推翻现有模板层与任务流的前提下，为中眠网 AI 编辑部建立“生成器 provider 层”。

本轮优先解决的问题是：

- 当前占位生成虽然已经模板化，但仍然直接发生在任务动作层里
- 如果后续接真实模型，现有任务保存逻辑仍会直接依赖具体生成实现
- provider、输出结构、任务记录之间还缺少一层稳定边界

本轮属于：

- 结构补强
- 可扩展性提升
- AI 工作流接口封装

## 2. 背景与上下文

本轮承接以下记录：

- `v0.13-prompt-template-layer`
- `v0.12-ai-editorial-workflow`

进入本轮之前，系统已经具备：

- 外置模板结构
- 结构化输入结构
- 模板驱动的模拟生成
- `AiTask + Content + Workflow` 的真实挂接边界

但还有一个关键问题没有解决：

- 当前“谁来生成内容”这件事仍然没有被抽象

也就是说，虽然模板层已经立住了，但任务动作层还是知道太多生成细节。这样未来接入真实模型时，仍有很大概率把现有动作层重新改一遍。

## 3. 问题分析

本轮的核心矛盾不是“有没有 mock generator”，而是：

- 任务系统能否在不关心具体生成器实现的前提下继续运行

如果这一点做不好，会出现几个问题：

1. 每增加一个生成器，就要改一次 action
2. 模拟生成器与未来真实模型生成器无法平滑替换
3. 生成结果结构没有统一标准，后续日志、版本、摘要都会再次散掉

所以本轮真正要建立的是一条清晰接口：

- `AiTask` 只关心 provider 输入与结果
- provider 只关心模板与结构化输入
- 模板层继续负责模板定义，不负责任务流

## 4. 候选方案比较

### 方案 A：继续让 action 直接调用占位生成函数

核心思路：

- 保持现在的模板层
- 只把占位生成函数名字改得更像 provider

优点：

- 改动最小
- 现有逻辑最容易延续

缺点：

- provider 只是表面抽象
- 任务流仍然直接耦合到具体生成实现
- 未来新增真实模型时，动作层仍然会被迫改动

结论：

- 不采用。

### 方案 B：建立统一 provider 接口 + 默认 placeholder provider

核心思路：

- 定义统一 provider 类型
- 定义统一生成请求与结果结构
- 当前先实现一个默认 placeholder provider
- action 只通过 provider 入口执行生成

优点：

- 生成器正式从任务动作层中解耦
- 后续可以直接新增 provider，而不重写任务系统
- 输出结构可以开始标准化

缺点：

- 需要同步调整类型、服务层和后台展示
- 需要明确当前阶段 provider 信息存放位置

结论：

- 本轮采用。

## 5. 最终决策

本轮最终采用“统一 provider 接口 + 默认 mock provider + 统一结果结构”的方案。

关键决策如下：

1. 不新增数据库字段，先复用 `AiTask.modelName` 记录当前 provider 标识。
2. 新建统一 provider 入口，由 action 通过 provider 执行生成，而不再直接调用旧生成逻辑。
3. provider 的输入统一为：
   - `taskType`
   - `template`
   - `input`
4. provider 的输出统一为：
   - `status`
   - `text`
   - `excerpt`
   - `prompt`
   - `meta`
   - `outputJson`
5. 当前默认 provider 为 `placeholder-mock`。
6. 模板层继续保留，provider 在模板层基础上产出结果，而不是反过来把模板逻辑塞回 action。

## 6. 具体实现

### 6.1 升级类型系统

修改：

- `features/admin/editorial/types.ts`

新增：

- `AiGenerationProviderId`
- `AiGenerationRequest`
- `AiGenerationMeta`
- `AiGenerationResult`
- `AiGenerationProvider`

作用：

- 为 provider 层建立统一输入和输出标准
- 让后续新增真实模型 provider 时不需要重新定义任务层结构

### 6.2 新增 provider 注册层

新增：

- `features/admin/editorial/provider.ts`

作用：

- 提供默认 provider 注册表
- 提供 `getAiGenerationProvider`
- 提供 `executeAiGeneration`

这一步的关键价值是：

- action 不再直接依赖某个具体生成实现

### 6.3 新增默认 placeholder provider

新增：

- `features/admin/editorial/providers/placeholder.ts`

作用：

- 在不接真实模型的前提下，继续保留当前 mock 能力
- 但现在它已经是一个标准 provider，而不是散落在任务流中的占位函数

provider 当前会统一输出：

- 生成正文文本
- 文本摘要片段
- provider / template 元信息
- 结构化 `outputJson`

### 6.4 调整模板输出能力

修改：

- `features/admin/editorial/templates.ts`

作用：

- 保留模板层对 prompt 与结构化 section 的输出职责
- 让 provider 基于模板结果再封装统一元信息与统一结果结构

也就是说，本轮没有推翻模板层，而是让模板层和 provider 层形成清晰分工。

### 6.5 调整任务动作层

修改：

- `features/admin/editorial/actions.ts`

作用：

- `GENERATE_PLACEHOLDER` 不再直接调用旧生成逻辑
- 改为调用 `executeAiGeneration`
- 成功后写入：
  - `prompt`
  - `outputText`
  - `outputJson`
  - `modelName`
- 继续沿用既有：
  - Content 草稿挂接
  - Workflow 记录
  - ContentVersion 记录

### 6.6 调整后台展示层

修改：

- `features/admin/editorial/server.ts`
- `app/admin/(panel)/ai-editorial/page.tsx`
- `app/admin/(panel)/ai-editorial/[id]/page.tsx`

作用：

- 列表页可以显示当前任务使用的 provider
- 详情页可以显示当前 provider 信息
- 让 `AiTask` 与 provider 的挂接关系变成可见状态，而不是只存在于内部逻辑里

## 7. 本轮优点

本轮完成后，AI 编辑部的生成环节相比之前有几个明显提升：

1. 当前生成逻辑已经不再直接写死在 action 中。
2. placeholder generator 已经成为标准 provider。
3. `AiTask` 已经可以明确记录当前使用的 provider。
4. 统一结果结构已经建立，后续日志、版本、provider 切换更容易接。
5. 不需要修改 Schema，就先把稳定接口层建立起来了。

## 8. 本轮缺点与代价

这轮仍然保留了几个现实边界：

1. provider 目前仍只有一个默认 mock 实现。
2. `modelName` 目前承担 provider 记录职责，语义上属于阶段性复用，并不是长期最理想字段。
3. 还没有建立独立的 provider 配置层或版本管理层。
4. 还没有 provider 级任务日志与重试机制。
5. 详情页当前只展示 provider 标识，没有做更完整的元信息浏览。

## 10. 本轮结论

本轮最重要的结论是：

中眠网 AI 编辑部已经从“模板驱动生成”，进一步推进成“可替换生成器的标准工作流”。

被证实的判断有：

- 不改 Schema 也可以先建立 provider 层
- provider 层和模板层可以形成稳定分工
- 当前 mock generator 完全可以作为未来真实 provider 的替身接口

以后默认应继承的经验是：

- 先稳定接口，再接真实模型
- provider 信息应尽量在任务层可见，而不是只藏在内部实现里

## 11. 下一轮建议

下一步最值得进入的方向有三个：

1. 真实模型接入前的配置层
   - 收益最高
   - 可以把 provider 默认值、模板策略、输出模式等配置收敛起来

2. 模板后台可配置化
   - 运营收益明显
   - 但结构复杂度会高于当前阶段

3. AI 任务日志 / 版本管理
   - 对调试与回溯最有帮助
   - 但适合在 provider 和配置层更稳定之后推进

如果只选一个，我会优先推荐：

- 真实模型接入前的配置层

因为当前 provider 接口已经成立，下一步最自然的工作就是把 provider 选择、默认行为和模板策略继续从 action 中抽出来。
