# 版本记录：v0.9-site-detail-pages-prisma

## 1. 本轮目标

本轮目标是把中眠网前台的三类详情页正式接到 Prisma 数据层，验证以下闭环是否已经成立：

- 后台真实录入
- 发布状态流转
- 前台只展示已发布内容

优先范围只包括：

- Content 详情页
- Term 详情页
- Brand 详情页

这一轮不做列表页全面接真实数据，不做后台继续扩展，也不做 AI 编辑部与 SEO 深挖。

## 2. 背景与上下文

本轮承接以下两期记录：

- `v0.7-schema-stage2-foundation`
- `v0.8-admin-crud-workflow`

在本轮开始前，中眠网已经具备：

- `Content + Term + Brand` 三主体数据底座
- `Workflow / ContentVersion / AiTask` 过程层
- 后台真实录入、保存、切状态、记录过程的最小闭环

但前台仍然停留在 demo 骨架阶段：

- 只有 `/knowledge`、`/terms`、`/brands` 的演示型列表页
- 没有真实详情路由
- 前台仍主要使用 `demo-data`

因此这轮的关键不是“再补几个页面”，而是验证数据底座是否真的打通到前台。

## 3. 问题分析

表面现象是：前台已经有栏目页，但没有真实详情页。

真正的核心矛盾是：

- 后台已经能真实录入内容
- 但前台还不能基于数据库直接读取与展示

这会导致一个危险假象：

- 看起来系统有后台、有前台
- 实际上生产链路仍然没有贯通

本轮需要解决的关键点有三个：

1. 详情页必须只读取真实 Prisma 数据，不能再回退到 demo 数据。
2. 未发布内容不能出现在公开前台。
3. `Content / Term / Brand` 三类详情页不能长成同一种文章模板，必须保留各自角色。

## 4. 候选方案比较

### 方案 A：直接在三个页面文件里各自写 Prisma 查询

核心思路：

- 在 `app/(site)/knowledge/[slug]`、`terms/[slug]`、`brands/[slug]` 各自直接写查询

优点：

- 开发速度快
- 文件少

缺点：

- 查询逻辑会分散
- 数据库不可用兜底会重复
- 后续接 `generateMetadata`、SEO、列表页真实数据时难以复用

放弃原因：

- 这轮虽然只做详情页，但已经进入“真实数据链路”阶段，查询层不适合继续散着写

### 方案 B：增加一层前台详情查询服务

核心思路：

- 新增 `features/site/details/server.ts`
- 统一封装三类详情页查询
- 统一处理数据库未配置、未发布不可见、关联查询与未来 SEO 复用

优点：

- 查询逻辑集中
- 详情页组件更专注于渲染
- 后续可直接复用到 `generateMetadata`、列表页真实接入、sitemap

缺点：

- 会新增一个服务文件
- 需要在服务层先把关联查询关系梳理清楚

最终选择原因：

- 更符合当前项目“统一内容底座 + 可持续扩展”的方向

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策如下：

1. 新增前台详情查询服务层，统一查询 `Content / Term / Brand`
2. 详情页只查询 `workflowStatus = PUBLISHED` 的数据
3. 未发布或不存在的数据统一走 `notFound()`
4. 数据库未配置时，不再回退 demo 数据，而是明确展示“真实数据未就绪”提示
5. 三类详情页保持不同角色：
   - `Content`：知识内容页
   - `Term`：定义 / 概念 / 关联节点页
   - `Brand`：企业名片 / 行业名录页

## 6. 具体实现

### 6.1 新增前台详情查询层

新增文件：

- `features/site/details/server.ts`

作用：

- 封装三类已发布详情查询
- 统一数据库不可用兜底
- 查询基础关联信息

具体能力：

- `getPublishedContentDetail(slug)`
- `getPublishedTermDetail(slug)`
- `getPublishedBrandDetail(slug)`

### 6.2 新增前台统一 404 兜底页

新增文件：

- `app/(site)/not-found.tsx`

作用：

- 当详情页 `slug` 不存在或资源未发布时，给出统一的前台兜底提示
- 明确说明“前台只展示已发布内容”

### 6.3 新增 Content 详情页

新增文件：

- `app/(site)/knowledge/[slug]/page.tsx`

展示内容：

- 标题
- 摘要
- 正文
- 发布时间 / 更新时间
- 分类 / 标签
- 相关词条
- 相关品牌
- 相关推荐

额外处理：

- 补充 `generateMetadata`
- 页面强制动态渲染，避免详情缓存妨碍状态验证

### 6.4 新增 Term 详情页

新增文件：

- `app/(site)/terms/[slug]/page.tsx`

展示内容：

- 词条名
- 一句话定义
- 详细解释
- 所属分类
- 相关词条
- 相关知识入口
- 更新时间

强调点：

- 页面不按文章结构组织，而是按“定义 + 解释 + 关联节点”组织

### 6.5 新增 Brand 详情页

新增文件：

- `app/(site)/brands/[slug]/page.tsx`

展示内容：

- 品牌名称
- 简介
- 详细介绍
- 分类 / 地区 / 城市
- 主营方向（优先读取 `tagline`，无则回退 `summary`）
- 官网
- 相关品牌
- 相关内容

强调点：

- 页面更像企业名片和行业名录，而不是内容文章页

## 7. 本轮优点

相比之前，这轮带来了几个明确改进：

1. “后台生产 -> 前台展示”的主链路首次真实打通
2. 未发布内容不再有暴露风险
3. 三类详情页已经明确分工，不会继续长成同一种模板
4. 查询层已经为后续 `metadata / sitemap / 列表页真实数据` 留出复用入口
5. 数据库未配置时不再偷偷回退 demo 数据，状态更诚实

## 8. 本轮缺点与代价

这轮也有明确代价和未完成项：

1. 列表页仍然主要是 demo 数据，没有一起切到 Prisma
2. `Content` 详情页的相关推荐仍是最小实现，只按分类 / 标签关系做基础推荐
3. `Term` 没有真正的“词条-词条”专属关系表，当前相关词条是按同分类 / 同标签近似推导
4. `Brand` 的“主营方向”目前优先读取 `tagline`，后台真实表单还没有把这个字段开放出来
5. 数据库未配置时只能显示“真实数据未就绪”，无法直接完成页面联调验证

这些妥协是当前阶段可接受的，因为本轮重点是先证明链路通了，而不是一次性补全所有细节。

## 9. 验证与结果

本轮实际做了以下验证：

1. `npm run typecheck`
2. `npm run build`
3. `npm run prisma:generate`

验证结果：

- 三类详情页路由均已成功进入构建产物
- 构建输出中可见：
  - `/knowledge/[slug]`
  - `/terms/[slug]`
  - `/brands/[slug]`

额外观察：

- 当前本地环境没有配置 `DATABASE_URL`
- 因此真实详情查询在运行时会进入“数据库未就绪”提示，而不是展示 demo 数据

## 10. 本轮结论

本轮最重要的结论是：

中眠网的数据底座已经不再只是后台可操作，而是已经具备前台真实展示的技术通路。

被证实的判断：

- `Content + Term + Brand` 三主体可以直接支撑前台三类详情页
- `PUBLISHED` 状态可以稳定承担前台公开可见的门槛
- 详情查询层值得保留，后续继续扩展不会浪费

应默认继承的经验：

- 前台真实数据页不要再回退 demo 数据
- 未发布内容必须通过查询层过滤掉，而不是交给页面层临时判断
- 中眠网不同资源类型的详情页必须保留明确角色差异

## 11. 下一轮建议

下一轮最值得做的 3 件事：

1. 前台列表页进一步接真实数据
   - 收益最高，因为用户能从栏目页直接进入真实详情页
2. SEO / metadata / sitemap
   - 风险最低，因为详情查询层已经准备好复用
3. AI 编辑部基础工作流
   - 需要更多后台联动场景后再做，优先级略低于前台列表与 SEO

如果只选一个，我会优先推荐：

- 前台列表页进一步接真实数据

原因是这样最能继续验证“统一内容底座 + 后台录入 + 前台访问路径”是否完整顺畅。
