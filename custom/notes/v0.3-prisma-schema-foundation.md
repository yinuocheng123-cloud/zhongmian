# 版本记录：v0.3-Prisma Schema 与真实内容底座初稿

## 1. 本轮目标

本轮目标是在 `v0.2` demo 骨架确认后，将中眠网推进到“有真实内容底座的可开发系统”阶段。

本轮重点是：

- 正式落地 Prisma Schema
- 建立统一内容底座与词条、品牌专项实体
- 建立内容状态流转模型
- 为 AI 编辑部与版本留痕预留最小可用数据结构
- 保持工程可运行，并通过 `prisma generate`、`typecheck`、`build`

本轮属于：

- 核心数据底座建设
- 可维护性提升
- 正式开发前的关键建模轮次

## 3. 问题分析

### 3.1 表面需求

用户要求优先完成：

- `Content`
- `Term`
- `Brand`
- `Category`
- `Tag`
- `Workflow`

并在必要时增加少量辅助实体，但不陷入复杂化。

### 3.2 真正的核心矛盾

这一轮真正要解决的不是“表怎么建”，而是：

- 如何让知识、专题、智库、标准、榜单、指数等未来模块尽可能复用统一内容模型
- 如何让词条、品牌既能独立展示，又能与内容互相连接
- 如何在没有完整权限系统的前提下先把发布状态流转跑通
- 如何让 AI 编辑部未来有承接位置，但当前不引入过重抽象

### 3.3 本轮关键判断点

本轮做出决策时，最关键的信息有三类：

1. 产品定位不是资讯站，而是知识基础设施
2. 一期后台需要内容流转和 AI 编辑部基础链路
3. 用户明确要求控制复杂度，避免一次建全

## 4. 候选方案比较

### 方案 A：为知识、专题、智库、标准分别建独立主表

**核心思路**

每个栏目独立建模，各自维护标题、正文、SEO、状态、分类等字段。

**优点**

- 初看很直观
- 每个栏目都能单独扩展

**缺点**

- 重复字段大量堆叠
- 后台表单和查询逻辑重复
- 后续改 SEO、状态流转、版本留痕时成本很高

**结论**

不选。与“统一内容底座”目标冲突。

### 方案 B：只保留一个 Content 万能表，词条和品牌也塞进去

**核心思路**

所有资源都统一放进 `Content`，通过类型字段区分。

**优点**

- 结构最统一
- 查询路径最简单

**缺点**

- 词条和品牌有明显不同的数据特征
- 品牌联系方式、地域、官网等字段会把 Content 拉得很散
- 词条定义、别名等结构也不适合完全挤进 Content

**结论**

不选。统一度高，但不符合业务差异。

### 方案 C：统一 Content 主体 + Term / Brand 专项实体扩展

**核心思路**

以 `Content` 承接文章、知识、专题、报告、标准、智库、榜单、指数等内容型资源；
以 `Term` 和 `Brand` 承接明显具有专项字段的实体；三者之间再通过分类、标签和关联关系互相连接。

**优点**

- 统一与差异化之间更平衡
- 后台管理模型清晰
- 未来前台详情页更容易拆分
- 扩展新栏目时更不容易推翻

**缺点**

- 关系设计稍多
- 工作流和 AI 任务需要支持多个目标实体

**结论**

最终采用本方案。

## 5. 最终决策

### 5.1 核心模型选择

本轮最终落地的模型为：

- `Content`
- `Term`
- `Brand`
- `Category`
- `Tag`
- `Workflow`
- `ContentVersion`
- `AiTask`

### 5.2 为什么不加更多辅助模型

本轮刻意没有加入：

- `SeoMeta`
- `AiTemplate`
- `MediaAsset`
- `User`
- `Permission`

原因是：

- 当前阶段最需要的是把真实内容底座搭稳
- SEO 字段直接放入主模型即可满足一期
- AI 模板和媒体库会让后台实现复杂度明显上升
- 用户明确要求暂不深入权限系统

## 6. 具体实现

### 6.1 Schema 总体结构

当前 Schema 分为四层：

1. 枚举层  
   `ContentType`、`WorkflowStatus`、`WorkflowTargetType`、`CategoryScope`、`AiTaskType`、`AiTaskStatus`

2. 核心资源层  
   `Content`、`Term`、`Brand`

3. 组织层  
   `Category`、`Tag`

4. 过程层  
   `Workflow`、`ContentVersion`、`AiTask`

### 6.2 各实体关系说明

#### Content

用于承载统一内容实体。

承载范围：

- 文章
- 知识内容
- 专题
- 指南
- 报告
- 智库内容
- 标准内容
- 榜单页面内容
- 指数页面内容

关联关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Term` 多对多
- 与 `Brand` 多对多
- 与 `Workflow` 一对多
- 与 `ContentVersion` 一对多
- 与 `AiTask` 一对多

#### Term

用于承载睡眠词条、术语、概念定义。

关联关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Content` 多对多
- 与 `Workflow` 一对多
- 与 `AiTask` 一对多

#### Brand

用于承载品牌、企业、机构展示信息。

关联关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Content` 多对多
- 与 `Workflow` 一对多
- 与 `AiTask` 一对多

#### Category

用于承载栏目分类体系，同时支持树状结构。

关联关系：

- 自关联形成父子分类
- 与 `Content`、`Term`、`Brand` 分别多对多

#### Tag

用于承载内容标签体系。

关联关系：

- 与 `Content`、`Term`、`Brand` 分别多对多

#### Workflow

用于承载内容状态流转和审核记录。

设计方式：

- 使用 `targetType` 区分目标资源类型
- 通过可空 `contentId` / `termId` / `brandId` 关联不同资源
- 记录 `fromStatus`、`toStatus`、`note`、`actorName`

这样做的原因是：

- 先保证对 `Content / Term / Brand` 三类核心资源的完整流转支持
- 后续如果增加 `Product` 等实体，可以平滑扩展新的可空关联字段

#### ContentVersion

用于内容版本留痕。

本轮只对 `Content` 开放，原因是：

- 内容类资源最先需要版本能力
- 词条和品牌未来如确实需要，可再复制同一思路扩展

#### AiTask

用于 AI 编辑部的任务占位。

设计方式：

- 可关联 `Content`、`Term`、`Brand`
- 记录任务类型、状态、提示词、输入输出与模型名

这样做的原因是：

- 为 AI 编辑部基础工作流预留真实挂点
- 不必现在就进入模板系统和复杂 Agent 编排

### 6.3 为什么这样建模

本轮建模遵循了三条原则：

1. **统一内容底座优先**  
   内容型资源统一收口到 `Content`

2. **专项实体保留业务特征**  
   词条和品牌不强行塞进 `Content`

3. **过程模型独立于展示模型**  
   工作流、版本、AI 任务都用独立模型承接，而不是把状态历史写死在主表里

### 6.5 Migration 建议

当前本地没有配置真实数据库，因此本轮没有执行正式 migration。

建议后续迁移顺序如下：

1. 初始化核心枚举与主表  
   `Content`、`Term`、`Brand`、`Category`、`Tag`

2. 初始化关联与索引  
   内容-分类、内容-标签、内容-词条、内容-品牌等多对多关系

3. 初始化过程表  
   `Workflow`、`ContentVersion`、`AiTask`

4. 导入 seed 演示数据  
   先让前后台可联调，再进入表单开发

建议 migration 名称可以按下面顺序创建：

- `init_content_foundation`
- `add_taxonomy_relations`
- `add_workflow_version_ai`

### 6.6 Seed 数据方案建议

本轮已新增 `prisma/seed.mjs`，采用最小演示数据策略。

当前 seed 包含：

- 3 个基础分类  
  睡眠知识 / 睡眠词库 / 睡眠品牌

- 2 个基础标签  
  产业观察 / 睡眠呼吸暂停

- 1 个词条  
  睡眠呼吸暂停

- 1 个品牌  
  眠域科技

- 1 条统一内容  
  睡眠产业图谱

- 对应内容版本记录
- 对应工作流记录
- 1 条 AI 任务示例

seed 的目的不是“填满数据”，而是：

- 让详情页开发时马上有真实结构可接
- 让后台表单开发时有真实数据可验证
- 让工作流和 AI 编辑部的关系尽早具象化

- SEO 字段先内嵌在主模型中，后续如果需要更统一的站点级 SEO 管理，可能要再抽象
- 目前没有正式 migration 文件，因为还没接真实数据库
- 还没有把前台列表和详情页接到真实查询

## 10. 本轮结论

本轮最重要的结论是：

中眠网已经从“前后台 demo 骨架”进入了“真实内容底座已落地”的阶段。

本轮确认的关键判断：

- `Content + Term + Brand` 的三层结构是当前最稳妥的方案

- 当前模型已经足以支撑下一步后台表单、发布流转和详情页接数据

## 11. 下一轮建议

下一步最值得进入的是：

1. 后台真实表单与发布流转  
   收益最高，因为已经有真实 Schema，可马上把内容、词条、品牌录入跑通

2. 详情页骨架接真实数据
