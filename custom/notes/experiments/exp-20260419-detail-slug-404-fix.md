# 实验记录：详情页中文 slug 404 修复

## 实验目标

只修复前台详情页访问链路中的 404 问题，不扩展任何功能，不调整业务结构。

目标范围：

- `/knowledge/[slug]`
- `/terms/[slug]`
- `/brands/[slug]`

## 实验设定

- 先验证路由结构、列表跳转和 Prisma 查询是否理论对齐
- 再用数据库实际数据和 HTTP 请求复现 404
- 只在详情查询入口做最小修复

## 关键变量

1. 列表页是否按 slug 跳转
2. 已发布记录是否确实存在 slug
3. 详情查询拿到的 slug 是否与数据库保存值一致

## 观察结果

1. 三类详情页路由本身存在，列表页跳转路径也都是：
   - `/knowledge/${slug}`
   - `/terms/${slug}`
   - `/brands/${slug}`
2. 数据库中 `Content / Term / Brand` 的已发布记录都存在 slug，且无空值、无重复。
3. 直接用 Prisma 按 `slug + PUBLISHED` 查询可以拿到数据。
4. HTTP 验证发现：
   - ASCII slug 可正常访问
   - 中文 slug 全部返回 404

这说明问题不在“有没有数据”，而在“详情查询使用的 slug 值和数据库实际 slug 值不一致”。

## 初步结论

本轮 404 的根因是中文 slug 在进入详情查询时仍带有 URL 编码，导致 Prisma 精确匹配失败，随后被 `notFound()` 误判成不存在。

## 是否值得继续

值得继续，但本轮只做最小修复即可：

1. 在详情查询入口统一做安全 `decodeURIComponent`
2. 保持未发布内容仍然按 `PUBLISHED` 条件返回 404
