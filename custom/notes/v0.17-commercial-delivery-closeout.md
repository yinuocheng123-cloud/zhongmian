# 版本记录：v0.17-commercial-delivery-closeout

## 1. 本轮目标

本轮目标是把中眠网从“功能已基本完成”继续收口到“可正式交付客户”的状态，重点解决三件事：

1. 后台列表页补上最小可用的批量运营能力
2. 长文本编辑区补上 Word / WPS 粘贴兼容与结构化渲染链路
3. 做一轮真实浏览器回归，并补齐客户交付时真正需要的后台使用说明与上线检查清单

本轮属于：

- 功能补充
- 可运营性补强
- 商用交付收口

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.16-operable-admin-backend.md`
- `custom/notes/v0.15-admin-min-auth.md`

进入本轮之前，项目已经具备：

- 前台公开访问主链路
- Content / Term / Brand 三主体
- 发布流转与前台只展示 `PUBLISHED`
- AI 编辑部模板层与 provider 层
- 后台最小鉴权
- 运营视角的后台仪表盘与基础管理页

但距离“可正式交付客户”还差两个明显短板：

1. 运营效率层还不够，列表页仍偏单条操作
2. 长文本编辑还停留在原始 textarea，不适合客户直接从 Word / WPS 粘贴内容

## 3. 问题分析

表面现象：

- 后台虽然能保存、发布、下线，但大量首批内容处理时会出现重复点击成本
- 文字内容虽然能录入，但从 Word / WPS 复制进来会丢结构或带入杂乱样式
- 批量操作初版基于 `form` 属性把 checkbox 绑定到表单外，浏览器原生可行，但在当前 Server Action 场景下不够稳

核心矛盾：

- 当前系统离“可运营”已经很近，真正缺的是运营效率与真实编辑体验，而不是继续加新模块

本轮主要瓶颈：

1. 批量操作要兼容现有单条快操作，又不能破坏当前列表页结构
2. 长文本要保留基础结构，又不能为了这件事推翻现有存储模型或引入过重编辑器体系
3. 需要在真实浏览器中验证，而不能只靠代码静态判断

## 4. 候选方案比较

### 方案 A：引入完整富文本编辑器与复杂批量表格

核心思路：

- 接入完整富文本编辑器
- 额外引入通用批量表格组件、全选、跨页选择等能力

优点：

- 能力完整
- 长期扩展空间大

缺点：

- 改动面明显过大
- 需要额外依赖与额外调试成本
- 不适合当前“交付收口”节奏

结论：

- 本轮不采用

### 方案 B：保留现有数据模型，用结构化文本标记 + 最小批量操作增强

核心思路：

- 长文本仍存字符串，但编辑器把 Word / WPS 粘贴转成统一结构化文本标记
- 前台详情页按统一结构化规则渲染
- 列表页用一个外层批量表单包裹，行内单条动作改为 `formAction` 分流

优点：

- 不推翻 Schema
- 不增加第三方依赖
- 对交付最关键的两个问题都能直接解决

缺点：

- 不是完整所见即所得富文本
- 结构化标记属于轻量方案，后续若追求更复杂排版仍可能升级

结论：

- 本轮采用

## 5. 最终决策

本轮最终采用“轻量结构化文本编辑 + 列表页最小批量操作 + 真实浏览器回归”的组合方案。

关键决策如下：

1. 不引入新依赖，避免在交付收口阶段增加依赖面
2. 用统一结构化文本组件贯通后台编辑与前台渲染
3. 列表页批量操作采用整页单表单，行内动作通过 `formAction` 分流，规避 checkbox 外挂表单在 Server Action 下不稳的问题
4. 真实浏览器回归优先验证客户最常用链路，而不是继续扩展功能面

## 6. 具体实现

### 6.1 结构化文本编辑与前台渲染

新增文件：

- `components/admin/structured-text-editor.tsx`
- `components/shared/structured-text-content.tsx`

修改文件：

- `components/admin/content-form.tsx`
- `components/admin/term-form.tsx`
- `components/admin/brand-form.tsx`
- `app/(site)/knowledge/[slug]/page.tsx`
- `app/(site)/terms/[slug]/page.tsx`
- `app/(site)/brands/[slug]/page.tsx`

作用：

- 后台长文本字段支持基础格式工具栏
- 支持 Word / WPS 粘贴时自动清洗冗余样式
- 将粘贴内容转成统一结构化文本标记
- 前台详情页按统一规则渲染标题、段落、列表、加粗、斜体

### 6.2 批量运营能力

修改文件：

- `features/admin/resources/actions.ts`
- `app/admin/(panel)/content/page.tsx`
- `app/admin/(panel)/terms/page.tsx`
- `app/admin/(panel)/brands/page.tsx`

作用：

- 新增批量状态切换服务端动作
- 列表页支持批量提审、批量发布、批量下线
- 统一用单表单 + `formAction` 处理批量与单条动作
- 发布 / 下线后同步触发前台与后台相关路径 revalidate

### 6.3 已有动作的稳定性补丁

延续修改文件：

- `features/admin/resources/actions.ts`
- `features/admin/editorial/actions.ts`

作用：

- 延续本轮之前已经发现的 `redirect` 被 `catch` 吞掉问题
- 确保保存 / 发布类动作在 Next.js 16 当前行为下仍能稳定跳转

### 6.4 交付文档

新增文件：

- `custom/delivery/admin-quick-guide.md`
- `custom/delivery/prelaunch-checklist.md`

作用：

- 为客户提供极简后台使用说明
- 为上线前最后收口提供可执行 checklist

## 7. 本轮优点

本轮的正向效果主要有：

- 后台真正具备了最小批量运营能力
- Word / WPS 粘贴终于进入“可用”状态，而不是停留在原始 textarea
- 前台详情页和后台编辑器围绕同一套结构化文本规则运行，更一致
- 批量动作与单条动作都保留在现有后台结构内，没有推翻已做成果
- 真实浏览器回归发现了批量操作表单结构不稳的问题，并在本轮内完成修复

## 8. 本轮缺点与代价

本轮也保留了明确边界：

- 结构化文本方案是轻量方案，不是完整富文本体系
- AI 编辑部的占位输出预览本轮没有完全统一到新编辑器组件中
- 批量操作目前是当前页勾选，不支持跨页全选
- 仍然存在部分测试残留内容，只是已经被压到 `OFFLINE` 或 `DRAFT`

这些代价在当前阶段是可接受的，因为本轮目标是交付收口，不是继续产品化后台。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`

   - 前台首页、列表页、详情页、未发布 404

5. 数据库状态复查：
   - 明显测试稿已被压到 `OFFLINE`

验证结论：

- 代码层通过
- 交互层关键主链路通过
- 交付层关键能力已经成形

## 10. 本轮结论

本轮最重要的结论是：

中眠网已经从“技术上可运行”进一步进入“客户拿到后可以实际运营”的状态。

被证实的判断：

- 不需要引入重型编辑器，也能把 Word / WPS 粘贴兼容做到可交付
- 不需要重构后台架构，也能把批量运营能力补到够用
- 真实浏览器回归是必要步骤，因为它确实帮我们抓出了 Server Action 表单结构问题

## 11. 下一轮建议

如果继续迭代，最值得做的 1~3 件事是：

1. 把 AI 编辑部输出区也统一到结构化文本编辑 / 预览体系
2. 为批量操作补一个“当前页全选”小能力
3. 在正式生产环境补后台账号环境变量与部署层配置校验
