# 版本记录：v0.25-reserved-extensions-complete

## 1. 本轮目标

本轮目标是把中眠网前期已经预留、但尚未完整落地的扩展模块一次性补齐，并且落到可运营、可发布、可交付的状态。

本轮属于：
- 功能补充
- 交付收口
- 可运营能力提升

之所以这一轮值得优先处理，是因为首页入口已经成立，但如果扩展栏目仍停留在结构化占位页，就会直接影响“行业入口 + 信任入口 + 商业入口”的完整性，也会削弱后台的长期运营价值。

## 2. 背景与上下文

本轮承接以下记录：
- `v0.22-knowledge-dev-timeout`
- `v0.23-homepage-industry-entry-completion`
- `v0.24-homepage-entry-route-closeout`

修改前的状态：
1. 首页入口结构已经完整，且核心入口不再 404。
2. `/trends`、`/events`、`/brand-progress`、`/rankings`、`/indexes`、`/standards`、`/think-tank`、`/cooperation` 已有可进入页面，但仍偏“结构化栏目页”。
3. 后台已经具备 Content / Term / Brand 三主体与 AI 编辑部能力，但扩展栏目的后台入口、列表、编辑与发布流转未完全接入。
4. AI 模板层和 provider 层已成立，可以继续沿用，不应推翻重做。
5. Prisma 当前以统一 Content 底座为核心，适合优先通过 `contentType + channelKey` 扩展，而不是另起多套模型。

已知约束：
- 不推翻现有 Content / Term / Brand / Workflow / AiTask 架构
- 不做复杂图表系统
- 不接真实 AI 模型
- 不做复杂 RBAC
- 不为了“优雅”重构已经稳定的主链路

## 3. 问题分析

表面现象：
- 扩展栏目已有路径，但缺少真实前台列表 / 详情能力
- 后台无法按栏目直接运营这些扩展内容
- 首页能点进去，但落地深度不够
- 交付文档没有同步覆盖新增栏目

核心矛盾：
- 中眠网已经不是简单的知识、词库、品牌三条主线，而是要成为真正的行业入口平台。
- 如果扩展栏目继续停留在预留层，平台定位会显得“说得很完整，实际只做了一半”。

主要原因：
1. 之前的阶段更强调先把主链路打通，扩展栏目只做了路由层面的收口。
2. 若为每个栏目单独建模、单独做后台，会明显增加复杂度和维护成本。
3. 项目已经有统一 Content 底座，因此最合理的方向是继续复用，而不是新增大量表。

## 4. 候选方案比较

### 方案 A：每个扩展栏目单独新增独立模型

核心思路：
- 为趋势、事件、品牌进展、榜单、指数、标准、智库分别创建独立 Prisma 模型

优点：
- 语义边界最清晰
- 每个栏目字段可完全定制

缺点：
- 模型数量暴涨
- 后台、前台、SEO、AI、seed 都要重复接线
- 对当前交付目标来说成本过高

放弃原因：
- 与“最小复杂度、最大收益”的原则不符

### 方案 B：全部继续停留在占位页，不接真实数据

核心思路：
- 保留现有栏目页结构，只优化文案和引导

优点：
- 改动最小

缺点：
- 不是真正的可运营栏目
- 客户拿到后无法直接使用
- 与本轮“预留扩展全部补齐”的目标冲突

放弃原因：
- 不满足正式商用交付标准

### 方案 C：优先复用 Content 模型，以 channelKey + contentType 扩展栏目

核心思路：
- 趋势、事件、品牌进展、榜单、指数、标准、智库全部优先落在统一 Content 底座上
- 用最小字段补齐事件、标准、品牌进展所需的结构化信息
- 后台通过动态栏目管理页复用统一 Content 管理能力

优点：
- 改动集中、结构统一
- 前后台、SEO、AI、seed 都能共用现有链路
- 便于长期运营和后续继续扩展

缺点：
- 某些栏目不是完全独立模型，字段表达需要做折中
- 榜单 / 指数 / 标准暂时仍属于“统一内容底座上的专项栏目”，不是深度产品化系统

最终选择原因：
- 这是最符合当前项目阶段和交付要求的方案

## 5. 最终决策

本轮最终采用方案 C。

关键决策如下：
1. 新增 `SiteChannelKey`，把扩展栏目纳入统一 Content 频道维度。
2. 趋势、事件、品牌进展、榜单、指数、标准、智库全部继续使用 Content。
3. 用最小字段补齐特殊栏目需要：
   - 事件：时间、地点、类型
   - 标准：版本号
   - 品牌进展：关联品牌
4. 前台新增动态栏目页和动态详情页，但仍遵守只展示 `PUBLISHED` 的公开逻辑。
5. 后台新增动态栏目管理页，复用 Content 的列表、新建、编辑、状态流转能力。
6. AI 编辑部不重写，只扩展模板和任务内容类型映射。
7. 合作页保持为正式说明页，不强行新增复杂表单系统。

## 7. 本轮优点

1. 预留扩展不再停留在结构化占位，而是进入真实可运营状态。
2. 统一 Content 底座被进一步验证，扩展成本可控。
3. 前台新增栏目具备真实列表、详情、SEO 与 sitemap 支持。
4. 后台新增栏目复用统一管理能力，运营学习成本低。
5. AI 编辑部与新增栏目接轨，不需要另起一套工作流。
6. 首页入口、栏目页、后台、AI、文档都完成了同步收口。

## 8. 本轮缺点与代价

1. 榜单 / 指数 / 标准当前仍挂在统一 Content 底座上，不是完全独立的数据产品模型。
2. 指数页当前仍以结构化文本解释为主，没有复杂图表系统。
3. 合作页当前是正式说明页，不是线索表单系统。
4. 某些栏目长期发展后，可能仍需要从统一 Content 里拆出更深层的专用模型。

这些都是当前阶段可接受的 trade-off，不构成阻断。

## 10. 本轮结论

本轮最重要的结论是：
1. 中眠网的预留扩展模块可以继续依托统一 Content 底座完成，而不必立即拆成多个复杂系统。
2. 通过 `channelKey + contentType + 最小字段扩展`，已经可以支撑趋势、事件、品牌进展、榜单、指数、标准、智库这些栏目进入可运营状态。
3. “首页入口完整”只有在“对应栏目可运营、后台可管理、AI 可协同、文档可交付”时才真正成立。

## 11. 下一轮建议

1. 先完成本轮最后的 `build` 与前后台页面回归。
2. 如验证通过，按栏目 / 后台 / AI 文档三个层次拆分清晰 commit 并 push。
3. 若后续继续迭代，优先考虑把 `/trends`、`/events`、`/brand-progress` 做更深的数据运营，而不是再扩新的预留栏目。
