# 版本记录：v0.18-delivery-ux-hardening

## 1. 本轮目标

本轮目标是把中眠网从“代码上可交付”进一步补强到“客户拿到后用起来更顺手、不容易抱怨”的状态。

本轮优先解决四类体验问题：

1. 后台操作反馈、危险操作确认、空状态与登录过期提示
2. Content / Term / Brand 表单的最小发布校验与 slug 反馈
3. structured-text-editor 的粘贴、清理格式与未保存提醒
4. AI 编辑部可读性，以及前台三条列表页的筛选反馈与无结果提示

本轮属于：

- 可运营性补强
- 交付体验补强
- 上线前细节收口

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.17-commercial-delivery-closeout.md`
- `custom/notes/v0.16-operable-admin-backend.md`
- `custom/notes/v0.15-admin-min-auth.md`

进入本轮之前，项目已经具备：

- 前台公开主链路
- Content / Term / Brand 真实后台录入与发布流转
- AI 编辑部基础工作流
- 后台最小鉴权
- 商用交付级的后台与交付文档

但从“客户真实使用”的角度看，仍有几类容易引发抱怨的细节问题：

1. 后台批量操作与发布动作反馈不够清楚
2. 危险动作缺少最小确认，误点成本偏高
3. 筛选无结果、列表空状态和登录过期提示不够统一
4. 编辑器虽然已支持 Word / WPS 粘贴，但还缺“清理格式”和未保存提醒

## 3. 问题分析

表面现象：

- 后台操作完成后虽然会跳转，但提示文案不够面向运营
- 批量动作一旦部分失败，运营很难快速判断到底成功了多少
- 空列表与筛选无结果都只显示一段文案，不够像正式后台
- 登录过期时会被拦到登录页，但缺少明确提示
- 编辑器支持粘贴，但缺少进一步整理格式和离开提醒

核心矛盾：

- 这不是“缺功能”，而是“最后一层运营体验没有压实”
- 如果不做这一层补强，客户虽然能用，但会频繁产生“是不是没成功”“是不是丢了”“为什么没有结果”的不确定感

本轮真正要解决的是：

1. 让关键动作有清晰反馈
2. 让高风险动作更难误触
3. 让空状态和异常状态有统一处理
4. 让编辑器和 AI 列表更像日常可用工具，而不是工程演示页

## 4. 候选方案比较

### 方案 A：引入完整 toast / modal / 富文本库

核心思路：

- 接入专门的 toast 系统、弹窗系统和更完整的富文本编辑器能力

优点：

- 能力更完整
- 交互更接近成熟 SaaS 后台

缺点：

- 改动面明显过大
- 会引入更多依赖和更多测试成本
- 不符合本轮“只做体验补强，不扩功能”的边界

结论：

- 本轮不采用

### 方案 B：沿用当前架构，在共享组件与现有页面上做最小补强

核心思路：

- 继续使用 `notice / error` 反馈链路
- 在现有按钮、列表页、表单页和登录页上补确认、反馈、空状态和提示
- 在 structured-text-editor 内部补最小能力，而不引入新编辑器体系

优点：

- 改动集中
- 与现有 Next.js + Server Actions 结构一致
- 收益高、风险低、验证成本可控

缺点：

- 不是完整 toast / modal 方案
- 批量反馈仍然基于页面提示条，而不是悬浮提醒

结论：

- 本轮采用

## 5. 最终决策

本轮最终采用“沿用现有架构，在共享层和关键页面上做最小体验补强”的方案。

关键决策如下：

1. 后台继续沿用 `notice / error` 顶部提示条，而不是新接 toast 系统
2. 危险动作通过客户端按钮上的最小 `confirm` 二次确认实现
3. 批量操作改成“成功数量 + 失败数量 + 失败原因摘要”反馈
4. 发布前校验与草稿保存校验分层处理，避免所有状态都被同一套强校验卡住
5. 登录过期继续走当前最小鉴权方案，只补“过期提示 + 自动清 cookie”
6. 编辑器继续沿用结构化文本方案，只补粘贴清洗、清理格式和未保存提醒

## 6. 具体实现

- `features/admin/resources/actions.ts`

### 6.2 空状态、无结果状态与登录过期提示

修改文件：

- `components/admin/admin-list-empty-state.tsx`

- `features/admin/auth/session.ts`
- `app/admin/login/page.tsx`

作用：

- Brand：提交审核 / 发布前必须补全简介与详细介绍

## 7. 本轮优点

本轮方案的正向效果主要有：

- 后台操作是否成功、成功多少、失败多少更清楚
- 高风险动作不容易误点直接生效
- 列表页不会再出现“只有空表格，没有下一步”的交付感缺口
- 登录过期不再像出错，而是像正常会话失效
- 草稿保存与正式发布的校验强度被区分开，运营体验更顺手
- 编辑器更接近真实客户从 Word / WPS 复制粘贴的使用方式
- AI 编辑部第一次打开时更容易理解“任务 → 生成 → Content”的关系
- 前台三条列表页更像行业入口，而不是普通列表页

## 8. 本轮缺点与代价

本轮仍然保留了一些明确边界：

1. 没有引入完整 toast 系统，反馈仍然是页面提示条
2. 危险操作使用的是最小 `confirm`，不是自定义弹窗
3. 删除能力当前仍然没有实际入口，因此本轮没有新增删除确认
4. 未保存提醒是浏览器级提醒，属于“最小可用”方案
5. 前台中屏 / 移动端本轮主要通过现有响应式结构收口，没有单独做视觉重排

这些代价在当前阶段是可接受的，因为本轮目标是“补使用体验”，不是继续产品化后台系统。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`

结果：

- `typecheck` 通过
- `lint` 通过
- `build` 通过

## 10. 本轮结论

本轮最重要的结论是：

中眠网已经从“可正式交付客户”进一步补强到了“客户拿到手后更顺手、更不容易因为细节问题产生抱怨”的状态。

被证实的判断有：

1. 不需要接入新的弹窗或 toast 体系，也能把后台关键反馈做清楚
2. 不需要换编辑器体系，也能把 Word / WPS 粘贴体验继续往前推进一层
3. 通过共享组件和现有页面局部收口，就足以明显提升交付感

## 11. 下一轮建议

如果后续继续迭代，最值得做的 1~3 件事是：

1. 为批量操作补“当前页全选 / 反选”
2. 把 AI 编辑部输出区也统一成结构化预览模式
3. 在真实浏览器中补一轮中屏与移动端截图回归，进一步确认首页、三列表页和三详情页的细节排版
