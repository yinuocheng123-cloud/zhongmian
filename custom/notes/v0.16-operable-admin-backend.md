# 版本记录：v0.16-operable-admin-backend

## 1. 本轮目标

本轮目标是继续完善中眠网后台，使其从“最小可用后台”提升到“正式可运营后台”的状态。

本轮优先解决四类问题：
- 仪表盘缺少运营视角，无法快速判断站点内容状态
- Content / Term / Brand 的后台管理体验不够统一，分类、标签、发布时间、前台跳转等信息不完整
- AI 编辑部虽然已经接入内容链路，但后台展示仍偏任务壳，缺少筛选、输入、输出和挂接视图
- 发布流转虽然已能工作，但缺少面向运营的快速操作与误发布防护

本轮属于：
- 功能补充
- 可运营性补强
- 上线前后台收口

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.15-admin-min-auth.md`
- `custom/notes/v0.14-provider-layer.md`

进入本轮之前，项目已经具备：
- 前台公开访问链路
- Content / Term / Brand 的真实数据底座
- 后台最小 CRUD 与发布流转
- AI 编辑部模板层与 provider 层
- 后台最小鉴权

但后台仍然存在一个明显落差：技术闭环已形成，运营闭环还不够顺手。后台可以“操作”，但距离“长期运营”还差一层面向运营人员的概览、效率和风险控制。

## 3. 问题分析

表面现象：
- `/admin` 更像数据入口，不像运营控制台
- Content / Term / Brand 表单已经能保存，但缺少分类标签、发布时间与前台联动入口
- 列表页能看数据，但不够适合审核、发布和快速判断风险
- AI 编辑部详情页与列表页的信息密度不够，运营看不出任务和内容流之间的连接状态

核心矛盾：
- 当前后台的“数据能动”已经成立，但“运营能判、能筛、能管、能查”的层还没有建立起来

主要瓶颈：
1. 运营所需信息分散在多个页里，没有在仪表盘和列表页聚合
2. 资源编辑表单缺少一期已经需要的运营字段
3. 发布动作没有面向运营的快捷入口
4. AI 编辑部缺少结构化输入 / 输出视图，导致任务链路不可读

## 4. 候选方案比较

### 方案 A：做一套更通用的后台抽象

核心思路：
- 抽象通用资源列表、通用资源表单、通用统计模块，再让 Content / Term / Brand / AiTask 全部接入

优点：
- 代码形态更统一
- 未来新增资源时扩展成本可能更低

缺点：
- 本轮改动面太大
- 很容易把“中眠网运营后台”做成“通用 CMS 骨架”
- 会引入额外抽象成本，影响当前上线收口节奏

结论：
- 本轮不采用

### 方案 B：延续现有结构，做针对性补强

核心思路：
- 保留当前模块边界
- 在资源服务层补齐分类、标签、发布时间、前台入口、可疑测试残留识别
- 在页面层补齐仪表盘、列表页与 AI 编辑部的运营视角

优点：
- 延续现有架构，不推翻 Schema
- 改动集中，验证成本可控
- 更符合“正式可运营”而不是“重新做后台”

缺点：
- 资源页之间仍有少量重复结构
- 暂时没有做更高阶的组件抽象

结论：
- 本轮采用

## 5. 最终决策

本轮最终采用“延续现有后台结构，在服务层与页面层做运营化补强”的方案。

关键决策如下：
1. 继续围绕 `features/admin/resources` 做三主体统一增强，而不是拆出新的后台框架
2. 仪表盘直接接真实 Prisma 数据，不再依赖 demo 统计
3. Content / Term / Brand 表单补齐一期运营必要字段：分类、标签、发布时间、主营方向等
4. 发布流转继续沿用当前 Workflow / ContentVersion 设计，但补上列表页快速动作
5. 对可疑测试残留内容增加发布前阻断与仪表盘提示，减少误发布风险
6. AI 编辑部继续延用现有任务流，不推翻模板层 / provider 层，只补足筛选、输入、输出、挂接展示

## 7. 本轮优点

相比之前，本轮带来的正向效果主要有：
- 后台首页终于具备运营视角
- 三主体管理页在字段、交互和动作层面更统一
- 发布流转效率更高，运营不必每次都进入详情页处理状态
- 测试残留误发布的风险被进一步压低
- AI 编辑部的“模板 -> 输入 -> 结果 -> Content 草稿”链路变得可读、可追踪
- 整体更接近“内容生产与发布中台”，而不是功能拼接后的后台集合

## 8. 本轮缺点与代价

这一轮也保留了明确边界：
- 仍然没有做多账号与 RBAC
- 列表页筛选仍是最小可用级别，没有做复杂组合查询与批量操作
- Content 的相关推荐仍是预留位，没有进入后台可编辑配置
- AI 编辑部还没有任务日志、失败重试历史、模板后台可配置化
- 可疑残留识别基于规则判断，不是绝对准确，仍需要人工复核

这些都是当前阶段可以接受的妥协，原因是本轮目标是“正式可运营”，不是“做完整后台产品”。

## 9. 验证与结果

本轮执行了以下验证：

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`

验证结果：
- 全部通过

确认成立的结论：

- 仪表盘、三主体后台、AI 编辑部页面在 Next.js 16 当前结构下可以正常编译
- 列表页快速动作与页面跳转没有引入新的类型错误和构建错误

## 10. 本轮结论

本轮最重要的结论是：
中眠网后台已经从“开发期可用后台”进一步进入“正式可运营后台”的阶段。

已经被证实的判断：
- 不推翻现有架构，也能把后台运营体验补到足够商用
- `Content / Term / Brand + Workflow / ContentVersion + AiTask` 这套底座足以继续支撑运营层建设
- AI 编辑部不需要先接真实模型，也可以先成为真实内容生产流程的一部分

后续应默认继承的经验：
- 优先在资源服务层统一能力，再在页面层按资源角色补表达
- 后台补强应始终服务于运营动作，而不是追求通用化抽象

## 11. 下一轮建议

下一步最值得继续做的 1~3 件事：

1. 做一轮真实浏览器人工回归
   - 收益最高
   - 可验证仪表盘、快速状态切换、AI 编辑部详情展示是否符合运营预期

2. 补最小批量运营能力
   - 风险最低
   - 例如批量下线测试残留、批量提交审核

3. 为生产环境补稳定性配置
   - 例如后台账号环境变量、部署后 cookie / HTTPS 验证、数据库备份策略
   - 这是正式商用前的最后一类非功能性工作
