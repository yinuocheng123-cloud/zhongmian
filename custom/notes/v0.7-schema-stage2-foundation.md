# 版本记录：v0.7-schema-stage2-foundation

## 1. 本轮目标

本轮目标不是继续做前台页面，也不是进入后台表单，而是正式确认并收口中眠网第二阶段的数据底座。

这一轮要解决的问题是：

- 把“统一内容实体模型”从方案层推进到可持续继承的工程基线
- 明确哪些模型已经够一期使用，哪些能力仍然只做预留
- 让后续后台、前台详情页与 AI 编辑部都可以直接复用同一套数据结构

本轮属于：

- 核心数据底座建设
- 结构基线确认
- 后续模块开发前的关键收口

## 2. 背景与上下文

### 2.1 承接关系

本轮承接：

- `custom/notes/v0.6-visual-tone-upgrade.md`
- `custom/notes/v0.3-prisma-schema-foundation.md`

其中：

- `v0.6` 解决的是首页与前台行业气质问题
- `v0.3` 已经给出过 Prisma Schema 的第一轮基础方案

这次并不是推翻 `v0.3` 重建，而是把那一轮的建模思路正式确认成“第二阶段数据底座”，并做最小必要修正。

### 2.2 修改前状态

修改前，仓库内其实已经存在一版较完整的 Schema：

- `Content`
- `Term`
- `Brand`
- `Category`
- `Tag`
- `Workflow`
- `ContentVersion`
- `AiTask`

也已经有对应的 seed 方案。

因此本轮真正要做的不是“从零新建”，而是回答三个问题：

1. 这套模型是否足够代表中眠网的一期数据底座
2. 这套模型是否还需要补更多辅助表
3. 哪些状态命名和依赖实现需要统一，避免后续开发出现割裂

## 3. 问题分析

### 3.1 表面需求

用户本轮明确要求：

- 正式设计并落地 Prisma Schema
- 覆盖 `Content / Term / Brand / Category / Tag / Workflow`
- 可选补充 `ContentVersion / SeoMeta / AiTask / MediaAsset`
- 给出模型关系、设计原因、一期必要字段与预留字段、migration 建议与 seed 建议

### 3.2 核心矛盾

本轮真正的核心矛盾不是“表够不够多”，而是：

- 既要保证统一内容底座成立
- 又不能把词条和品牌强行塞回统一内容模型，导致业务特征被抹平
- 同时还要控制复杂度，避免为了未来可能的扩展，过早把 Schema 设计得过重

### 3.3 本轮关键判断

本轮最重要的判断有三点：

1. `Content + Term + Brand` 的三主体结构仍然是当前最稳妥方案
2. `SeoMeta` 与 `MediaAsset` 目前仍然不值得单独抽表
3. `WorkflowStatus` 需要统一到更贴近业务语言的命名，便于后续后台流转直接复用

## 4. 候选方案比较

### 方案 A：继续沿用现有 Schema，不做任何调整

**核心思路**

承认当前 Schema 已经够用，不再做任何收口，只在说明文档里解释。

**优点**

- 改动最小
- 风险最低

**缺点**

- 已有代码里仍然存在旧状态命名
- 第二阶段的“正式基线”不够明确
- 后续后台开发时容易继续出现术语不一致

**结论**

不选。因为这轮的重点就是把基线收口，而不是继续模糊过渡。

### 方案 B：继续扩展更多辅助模型

**核心思路**

在当前 Schema 上再增加 `SeoMeta`、`MediaAsset`、更复杂的版本模型或更完整的 AI 模板模型。

**优点**

- 未来理论扩展位更多

**缺点**

- 一期成本明显上升
- 后台和前台还没开始真正消费这些模型
- 会把本轮从“基础底座确认”变成“过度设计”

**结论**

不选。当前阶段最重要的是可扩展，但不是一次建全。

### 方案 C：保留现有核心模型，只做最小必要收口

**核心思路**

- 保留 `Content / Term / Brand / Category / Tag / Workflow`
- 保留 `ContentVersion / AiTask` 作为轻量预留
- 暂不新增 `SeoMeta / MediaAsset`
- 统一状态命名，让后台和后续流程直接复用

**优点**

- 延续既有正确思路
- 复杂度受控
- 足够支撑下一步后台表单、前台详情页和 AI 编辑部基础流转

**缺点**

- `Workflow` 仍采用多可空外键策略，后续若资源类型继续变多，需要再谨慎扩展
- `ContentVersion` 目前只覆盖 `Content`

**结论**

本轮最终采用该方案。

## 5. 最终决策

### 5.1 核心模型

本轮正式确认以下模型作为中眠网第二阶段的数据底座：

- `Content`
- `Term`
- `Brand`
- `Category`
- `Tag`
- `Workflow`
- `ContentVersion`
- `AiTask`

### 5.2 状态流转命名

本轮统一确认发布流转主状态为：

- `DRAFT`
- `PENDING_EDIT`
- `PENDING_REVIEW`
- `PUBLISHED`
- `OFFLINE`

其中：

- 用户要求的一期主流程是“草稿 → 待审核 → 已发布 → 已下线”
- 当前保留 `PENDING_EDIT`，因为它对后台编辑流转是有价值的中间态，而且已有结构已经围绕它展开
- 原有 `UNPUBLISHED` 命名调整为 `OFFLINE`，因为它更符合“已下线”的业务语义

### 5.3 是否新增辅助模型

本轮明确不新增以下模型：

- `SeoMeta`
- `MediaAsset`

原因是：

- 当前 SEO 字段直接放在主模型中已经足够一期使用
- 媒体资源库在前后台都还没有真实消费路径，现在抽表会徒增复杂度

## 6. 具体实现

### 6.1 实际修改文件

- `prisma/schema.prisma`
- `features/admin/resources/constants.ts`
- `features/admin/resources/utils.ts`
- `AGENTS.md`
- `custom/notes/v0.7-schema-stage2-foundation.md`

### 6.2 各文件作用

#### `prisma/schema.prisma`

正式确认并保留第二阶段核心模型，统一 `WorkflowStatus` 中“已下线”状态的命名为 `OFFLINE`。

#### `features/admin/resources/constants.ts`

同步后台资源管理常量中的状态标签、筛选项与样式映射，避免与 Prisma Schema 脱节。

#### `features/admin/resources/utils.ts`

同步资源表单状态动作到 `WorkflowStatus` 的映射逻辑，保证后续表单流转直接对齐新命名。

#### `AGENTS.md`

更新“当前最新一期正式记录”路径，便于后续迭代读取最新基线。

#### `custom/notes/v0.7-schema-stage2-foundation.md`

沉淀本轮设计判断、保留与放弃的方案、字段分层、迁移建议和 seed 策略，作为后续继续开发的正式依据。

### 6.3 模型关系说明

#### `Content`

统一承载文章、知识、专题、指南、报告、标准、榜单、指数、智库等内容型资源。

关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Term` 多对多
- 与 `Brand` 多对多
- 与 `Workflow` 一对多
- 与 `ContentVersion` 一对多
- 与 `AiTask` 一对多

#### `Term`

独立承载睡眠词条、术语、概念定义，是知识基础设施的一部分，不等同于文章。

关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Content` 多对多
- 与 `Workflow` 一对多
- 与 `AiTask` 一对多

#### `Brand`

独立承载品牌、企业、机构信息，服务于信任入口与商业入口，不等同于内容页。

关系：

- 与 `Category` 多对多
- 与 `Tag` 多对多
- 与 `Content` 多对多
- 与 `Workflow` 一对多
- 与 `AiTask` 一对多

#### `Category`

承担栏目与树形分类结构，支持 `Content / Term / Brand` 共享。

#### `Tag`

承担跨实体标签体系，支持 `Content / Term / Brand` 共享。

#### `Workflow`

承担发布流转与审核记录。

设计方式：

- 用 `targetType` 区分目标资源
- 用 `contentId / termId / brandId` 三个可空外键承接三类核心实体

这样做的原因是：

- 先确保三条主线可用
- 暂不引入更重的多态关联抽象

#### `ContentVersion`

承担统一内容实体的版本记录，目前只覆盖 `Content`，优先满足知识/专题/标准/智库等内容编辑需求。

#### `AiTask`

承担 AI 编辑部的任务挂点，可关联 `Content / Term / Brand`，满足后续“资料录入 → AI 生成 → 编辑 → 审核”流程的最小承接需求。

## 7. 本轮优点

- 统一内容底座没有被推翻，后续开发成本更可控
- 词条与品牌保持独立，符合中眠网的知识入口与信任入口定位
- `Workflow` 命名与业务语义更贴近，后续后台流转更顺手
- `ContentVersion` 和 `AiTask` 继续保持轻量可用，不会阻塞下一步
- 本轮没有为了“未来可能需要”而继续堆更多抽象

## 8. 本轮缺点与代价

- `Workflow` 仍然依赖多可空外键，不是最理想的多态建模
- `ContentVersion` 只覆盖 `Content`，如果未来词条和品牌也强依赖版本记录，还要补模型
- `SeoMeta` 与 `MediaAsset` 暂未独立抽表，未来若 SEO 运营或媒体管理变复杂，仍可能重构
- 当前只是结构底座确认，不代表后台、前台、AI 编辑部已经真实打通

## 10. 本轮结论

本轮最重要的结论是：

中眠网第二阶段的数据底座不需要推翻重来，当前最正确的做法是保留 `Content + Term + Brand` 的三主体结构，并以最小必要调整把它正式收口成可继承基线。

本轮确认应默认继承的经验是：

- 内容型资源优先统一到 `Content`
- 词条和品牌必须保持独立
- 工作流、版本、AI 任务应当独立于展示模型存在
- 一期先保证结构正确，不为了未来预想过度抽象

## 11. 下一轮建议

下一步最值得进入的方向有三件：

1. 后台真实表单与发布流转  
   收益最高，因为当前 Schema 和状态流转已经稳定，可以直接承接后台最小 CRUD。

2. 前台详情页接真实数据  
   风险较低，能快速把词条、知识、品牌从 demo 页面推进到真实数据读取。

3. AI 编辑部基础工作流  
   适合放在后台录入和状态流转跑通之后，再把 `AiTask` 真正接入流程。
