# 版本记录：v0.19-commercial-final-closeout

## 1. 本轮目标

本轮目标是把中眠网从“已可交付”进一步收口到“商用级正式交付”的代码状态，重点补齐最后一批仍值得落代码的细节：

1. 给后台三类主体补上真正可用的删除闭环
2. 把状态切换与批量操作的服务端跳转链路补稳
3. 让 `lint` 校验在当前仓库结构下稳定可执行
4. 补齐交付文档中关于删除与风险控制的说明

本轮属于：
- 交付收口
- 运营体验补强
- 验证与交付辅助完善

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.18-delivery-ux-hardening.md`
- `custom/notes/v0.17-commercial-delivery-closeout.md`

进入本轮之前，项目已经具备：

- 前台行业入口结构
- Content / Term / Brand 三主体
- 前台列表页与详情页真实 Prisma 链路
- 发布流转与前台只展示 `PUBLISHED`
- AI 编辑部模板层、provider 层与草稿挂接
- 后台最小鉴权
- 后台批量发布 / 下线 / 提审
- Word / WPS 粘贴兼容与结构化渲染

但从“客户长期运营”的视角看，仍有几个最后的缺口：

1. 删除还没有真正的后台闭环，只能靠手工下线和数据库清理
2. 状态切换类 action 虽然能跑，但仍值得把 redirect 异常兜底做得更稳
3. `npm run lint` 在当前仓库里会因为扫描非源码目录而卡住，不利于正式交付
4. 交付文档里还缺少对“删除规则”的明确说明

## 3. 问题分析

表面现象：

1. 用户已经明确要求“删除成功/失败提示”和“危险操作二次确认”，但当前后台只有发布 / 下线 / 批量动作，没有真正的删除入口
2. `features/admin/resources/actions.ts` 中的列表页状态切换 action 仍然存在把 redirect 当普通错误吞掉的风险
3. `eslint` 默认扫整仓时，会把 `custom/`、`test-results/` 等目录也纳入检查范围，导致 `npm run lint` 不稳定
4. 虽然 v0.18 已经补了大量体验层能力，但交付文档对删除边界没有同步补齐

核心矛盾：

- 本轮不需要新模块，也不需要大重构；真正缺的是“最后一层可运营细节”
- 如果这些细节不补，客户虽然能用，但会在删除、验证和上线收口时频繁卡住

## 4. 候选方案比较

### 方案 A：继续保持“只下线，不删除”

核心思路：
- 维持现状，只让后台支持下线和草稿，不提供删除入口

优点：
- 改动最少
- 没有新的数据删除风险

缺点：
- 与当前交付目标不符
- 测试残留和重复稿件只能越积越多
- 客户长期运营时会明显抱怨

结论：
- 本轮放弃

### 方案 B：补最小删除闭环，只允许删除 DRAFT / OFFLINE

核心思路：
- 不做批量删除，不做复杂回收站
- 只在 Content / Term / Brand 编辑页底部提供删除入口
- 只允许删除 `DRAFT` / `OFFLINE`
- 已发布或待审核内容要求先下线或转回草稿
- 依赖现有 Prisma 级联关系清理 Workflow / ContentVersion / AiTask

优点：
- 改动局部
- 风险清晰
- 直接解决商用运营中的残留清理问题

缺点：
- 还不是完整的“回收站 / 恢复”能力
- 删除只能单条执行，不能批量

结论：
- 本轮采用

## 5. 最终决策

本轮最终采用“最小删除闭环 + redirect 兜底补稳 + lint 范围收口 + 交付文档同步”的方案。

关键设计决策：

1. 删除只允许发生在 `DRAFT` / `OFFLINE`
2. 删除入口只放在三类编辑页底部，不扩散到列表页
3. 删除使用现有 `FormSubmitButton`，沿用统一确认交互
4. 新增一组 “Safe” 状态切换 action，避免推翻现有调用面
5. `eslint.config.mjs` 只补交付相关的忽略目录，不改业务代码

## 6. 具体实现

### 6.1 资源删除闭环

修改文件：
- `features/admin/resources/actions.ts`
- `app/admin/(panel)/content/[id]/page.tsx`
- `app/admin/(panel)/terms/[id]/page.tsx`
- `app/admin/(panel)/brands/[id]/page.tsx`

作用：
- 新增 `deleteResourceAction`
- 仅允许删除草稿或已下线内容
- 已发布或待审核内容给出明确错误提示
- 删除后回到列表页并带成功提示
- Prisma 通过既有 `onDelete: Cascade` 自动清理流程、版本和 AI 挂接

### 6.2 状态切换与批量操作安全版 action

修改文件：
- `features/admin/resources/actions.ts`
- `app/admin/(panel)/content/page.tsx`
- `app/admin/(panel)/terms/page.tsx`
- `app/admin/(panel)/brands/page.tsx`

作用：
- 新增 `changeResourceWorkflowStatusActionSafe`
- 新增 `changeBulkResourceWorkflowStatusActionSafe`
- 对 redirect error 单独放行，避免把成功跳转误当失败处理
- 批量操作反馈继续保留“成功数量 + 失败数量 + 摘要原因”

### 6.3 lint 稳定性收口

修改文件：
- `eslint.config.mjs`

作用：
- 明确忽略 `custom/**`
- 明确忽略 `test-results/**`
- 忽略 `.next-dev*.log`

这样可以保证 `npm run lint` 面向源码目录稳定执行，不被交付记录、实验文件和本地回归产物拖住

### 6.4 交付文档补齐

修改文件：
- `custom/delivery/admin-quick-guide.md`
- `custom/delivery/prelaunch-checklist.md`

作用：
- 在后台使用说明中补充“删除与风险控制”
- 在上线检查单里补充“仅草稿 / 已下线内容可删除”

## 7. 本轮优点

本轮带来的主要正向效果：

1. 后台现在真正具备“删草稿、清残留”的闭环
2. 删除能力有边界，不会误删已发布内容
3. 列表页状态切换与批量操作的跳转链路更稳
4. `npm run lint` 终于回到稳定可用状态，适合作为正式交付前校验
5. 交付文档与实际后台行为保持一致

## 8. 本轮缺点与代价

这一轮仍然保留了明确边界：

1. 没有做批量删除
2. 没有做回收站或恢复机制
3. 删除仍然是“最终删除”，不是软删除
4. 本地 `next dev` 在当前 Windows 环境下对部分页面请求存在不稳定现象，因此页面回归需要结合构建结果、数据库验证和 dev 日志综合判断

这些代价在当前阶段是可接受的，因为本轮目标是“正式商用交付”，不是继续产品化后台系统。

## 9. 验证与结果

本轮完成了以下验证：

1. `npm run typecheck` 通过
2. `npm run lint` 通过
3. `npm run build` 通过
4. 本地 PostgreSQL 已恢复，`DATABASE_URL` 可用，Prisma 读写链路正常
5. 通过 Prisma 临时数据验证了：
   - 可创建 DRAFT 的 Content / Term / Brand
   - 可创建 PUBLISHED 的中文 slug Content
   - 可创建并挂接 AiTask 到 Content 草稿
6. 本地 `next dev` 日志确认前台与后台关键页面均已被真实访问并返回状态码

需要如实说明的点：

- 当前 Windows 本地环境里，`next dev` 对部分 `Invoke-WebRequest / curl` 请求的客户端响应不稳定，客户端会出现超时，但服务端日志中能看到页面已完成响应
- 因此这轮“真实页面回归”是基于：
  - 构建通过
  - 数据库真实链路
  - `next dev` 页面访问日志
  - 后台与前台页面实际可达性
  的综合判断完成的

## 10. 本轮结论

本轮最重要的结论是：

中眠网已经补齐到“商用级正式交付”的代码状态。

被证实的判断：

1. 在不改 Schema 主结构的前提下，仍然可以把后台运营闭环补完整
2. 删除能力不需要做成复杂系统，只要边界清楚，就足以满足当前商用交付
3. 交付前的验证脚本同样属于正式交付质量的一部分，lint 的稳定性必须被视为交付项而不是附属项

## 11. 下一轮建议

如果后续继续迭代，最值得做的 1~3 件事是：

1. 为删除能力补最小“软删除 / 回收站”设计
2. 为批量操作补“当前页全选 / 反选”
3. 在正式部署环境做一轮更稳定的真实浏览器截图回归，补齐交付截图档案
