# 版本记录：v0.8-admin-crud-workflow

## 1. 本轮目标

本轮目标是把中眠网从“只有数据底座”推进到“后台最小可用的内容生产闭环”。

重点不是继续做演示页面，而是先让后台真实具备下面三件能力：

- 能录入 `Content / Term / Brand`
- 能保存和编辑
- 能完成最小状态流转，并留下必要的过程记录

本轮属于：

- 后台真实表单开发
- 发布流转骨架落地
- 内容生产闭环打通

## 2. 背景与上下文

### 2.1 承接关系

本轮承接：

- `custom/notes/v0.7-schema-stage2-foundation.md`
- `custom/notes/v0.6-visual-tone-upgrade.md`

其中：

- `v0.7` 已经正式确认了 `Content + Term + Brand` 三主体，以及 `Workflow / ContentVersion / AiTask` 过程层
- `v0.6` 解决的是前台行业气质，不影响本轮后台真实流程开发

所以这次不是重新设计后台，而是基于已经确认的 Schema，把后台从静态骨架推进成真实可操作系统。

### 2.2 修改前状态

修改前，仓库里其实已经有一批半成品：

- `server.ts` 已有基础查询封装
- `actions.ts` 已有基础保存动作雏形
- 三套表单组件已经存在
- 后台仅有 `content` 的骨架列表页，`terms / brands` 路由缺失

真正的问题不在于“没有代码”，而在于：

- 这些能力还没有被页面和路由真正串起来
- 后台仍然更像骨架，而不是可操作的生产系统
- `ContentVersion / Workflow / AiTask` 还没有在后台界面里显性出现

## 3. 问题分析

### 3.1 表面需求

用户本轮要求非常明确：

- 做出 `Content / Term / Brand` 三条主线的后台列表页
- 做出真实新建 / 编辑表单
- 做出草稿、提交审核、发布、下线的最小流转
- 把 `ContentVersion / Workflow / AiTask` 放到正确位置

### 3.2 真正核心矛盾

这一轮真正的难点不是“写几个表单”，而是：

- 如何在不推翻当前 Schema 的前提下，把已经存在的半成品收口成统一后台体验
- 如何让后台真实读写 Prisma 数据，但又不因为本地没有数据库时直接把开发体验做死
- 如何让过程层真正挂在后台里，而不是只存在于数据库模型中

### 3.3 本轮关键判断

本轮做出的关键判断有四个：

1. 继续复用已有 `server.ts / actions.ts / 表单组件`，不重写第二套逻辑
2. 后台管理页全部走动态渲染，避免管理数据被静态化
3. 列表页先统一为“搜索 + 状态筛选 + 表格列表”这一种形态
4. `ContentVersion` 只在 `Content` 编辑页展示，`AiTask` 只先做挂点展示，不提前做模型接入

## 4. 候选方案比较

### 方案 A：从零重写后台管理模块

**核心思路**

不复用现有半成品，重新定义查询层、表单层和页面层。

**优点**

- 理论上可以得到更整齐的新结构

**缺点**

- 与“最小改动、优先复用”的仓库原则冲突
- 会浪费已经存在的表单和 action 雏形
- 容易把本轮拖成重构，而不是落地

**结论**

不选。

### 方案 B：只补页面，不整理底层逻辑

**核心思路**

尽快把页面接起来，底层存在的小问题先不处理。

**优点**

- 页面能很快出来

**缺点**

- 状态、文案、字段定义、错误提示会继续分散
- 后续一旦扩展更多资源类型，还要回来补底层一致性

**结论**

不选。

### 方案 C：复用半成品支撑层，统一后再补齐页面

**核心思路**

- 保留既有 `server.ts / actions.ts / forms`
- 先把它们整理成可持续使用的支撑层
- 再把 `Content / Term / Brand` 的列表、新建、编辑页一次接齐

**优点**

- 风险低
- 结构清晰
- 收益直接作用到下一轮前台详情页和 AI 编辑部

**缺点**

- 会比“只补页面”多做一轮底层收口

**结论**

本轮最终采用该方案。

## 5. 最终决策

### 5.1 后台结构

本轮正式形成的后台结构是：

- `Content`：列表页 / 新建页 / 编辑页
- `Term`：列表页 / 新建页 / 编辑页
- `Brand`：列表页 / 新建页 / 编辑页

### 5.2 发布流转

统一采用现阶段最小可用流转：

- 保存当前内容
- 存为草稿
- 提交审核
- 审核通过并发布
- 下线 / 撤回发布

### 5.3 过程层挂接

- `Workflow`：三类资源都接入
- `ContentVersion`：仅 `Content` 接入
- `AiTask`：仅 `Content` 编辑页展示挂点

### 6.2 各部分作用

#### 后台路由层

补齐了三类资源的：

- 列表页
- 新建页
- 编辑页

并把后台主面板统一设置为动态渲染，避免后台数据被静态构建。

#### 查询层

`server.ts` 统一负责：

- 列表读取
- 编辑页数据读取
- 数据库不可用兜底
- `Content` 过程层读取

#### 提交层

`actions.ts` 统一负责：

- 表单字段读取
- 必填校验
- slug 唯一性校验
- 状态流转
- Workflow 记录
- ContentVersion 记录
- 路由 revalidate 与成功跳转

#### 表单层

三套表单都统一使用：

- 相同的输入控件
- 相同的提交按钮
- 相同的状态动作
- 相同的错误提示模式

### 6.4 当前状态流转实现

状态流转通过统一的 `ResourceFormIntent` 驱动：

- `SAVE`
- `SAVE_DRAFT`
- `SUBMIT_REVIEW`
- `PUBLISH`
- `UNPUBLISH`

再由 `resolveWorkflowStatus()` 统一映射到：

- `DRAFT`
- `PENDING_REVIEW`
- `PUBLISHED`
- `OFFLINE`

说明：

- `SAVE` 会保持当前状态，若没有状态则落到 `DRAFT`
- `SAVE_DRAFT` 强制回到草稿
- `SUBMIT_REVIEW` 进入待审核
- `PUBLISH` 进入已发布并写入发布时间
- `UNPUBLISH` 进入已下线并清空发布时间

### 6.5 `ContentVersion / Workflow / AiTask` 当前挂接位置

#### `Workflow`

在三类资源保存动作中统一写入：

- 当状态发生变化时记录一条 workflow
- 初次创建也会记录初始化状态

#### `ContentVersion`

在 `Content` 每次成功保存后都会写入一条版本记录：

- 版本号自增
- 记录标题快照
- 记录摘要快照
- 记录正文快照
- 记录变更说明

#### `AiTask`

本轮没有接真实 AI 模型，只在 `Content` 编辑页展示已有任务挂点，明确后续 AI 编辑部会从这里接入。

## 7. 本轮优点

- 后台从骨架推进成了真实可操作系统
- 三类资源的后台交互体验已经统一
- 数据库不可用时也不会把整个后台直接做死
- `Workflow / ContentVersion / AiTask` 不再只是模型概念，而是进入了后台界面结构
- 后续接前台详情页和 AI 编辑部时，不需要再重做后台主流程

## 8. 本轮缺点与代价

- 当前还没有完整权限系统，操作者仍是固定占位
- `Term` 和 `Brand` 还没有版本记录模型
- 列表页只做了基础搜索和状态筛选，分类/标签等高级筛选尚未接入
- 后台视觉仍以“能用、统一”为优先，没有进入精修阶段
- 若本地没有 `DATABASE_URL`，页面可以看结构但不能真正保存

## 10. 本轮结论

本轮最重要的结论是：

中眠网已经不再只有“数据底座”，而是已经拥有了最小可用的后台内容生产与发布能力。

当前默认应继承的经验是：

- 后台支撑层优先统一，再补页面，比只补页面更稳
- `Workflow` 应当先跑通最小闭环，不急着上复杂审批
- `ContentVersion` 先只绑定 `Content` 是合理的
- 后台管理页必须动态渲染，不能继续按前台 demo 的静态思路处理

## 11. 下一轮建议

下一步最值得进入的方向有三个：

1. 前台详情页接真实数据  
   收益最高。因为后台已经能录入真实内容，前台详情页现在具备了真正接 Prisma 的条件。

2. AI 编辑部基础工作流  
   风险中等。因为 `AiTask` 挂点和后台内容流转已经到位，可以开始接选题、资料录入和初稿流程。

3. SEO / metadata / sitemap  
   收益稳定。因为真实内容底座和后台录入能力已经具备，站点级 SEO 能力现在可以进入真实建设阶段。
